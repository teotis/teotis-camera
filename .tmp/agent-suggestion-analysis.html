2. 五项建议逐一判定

建议 1：CameraXCaptureAdapter 拆分为 Capture/Photo/Video/Config 模块

不建议落地

判定理由：拆分建议违反项目架构契约

AGENTS.md 明确定义了四层架构：Mode Plugin → Session Kernel → Device Adapter → Media Pipeline。CameraXCaptureAdapter 是 Device Adapter 层，其职责就是将抽象的设备请求翻译为 CameraX/Camera2 平台调用。

展开：为什么拆分不合理
  1. 前 1295 行是工具函数而非类方法：文件中大量 internal fun 是设备能力检测（detectZoomRatioCapability、detectPreviewBrightnessRange）、视频规格解析（buildVideoSpecConstraints）、实时照片处理（materializeLivePhotoSidecar）等。这些已经是独立函数，拆分只是移动文件位置，不改变耦合关系。
  2. 类本身约 2075 行、44 个方法：对于一个 CameraX 适配器来说偏大但不异常。CameraX API 本身就有 Preview、ImageCapture、VideoCapture、Camera2Interop 等多个 UseCase，适配器需要统一管理它们的绑定和生命周期。
  3. 拆分引入的复杂度：Capture/Photo/Video/Config 四个模块之间需要共享 CameraProvider、Camera 实例、PreviewView、生命周期绑定状态。拆分后需要一个协调层来管理这些共享状态，反而增加耦合。
  4. 已有边界：项目的 core/device 模块已经定义了 DeviceCapabilities、DeviceCommand、DeviceEvent 等抽象合约。CameraXCaptureAdapter 作为这些合约的实现，内部职责已经由合约边界约束。

如果确实需要优化：可以将前 1295 行的工具函数提取到独立的 CameraXCapabilityDetector.kt 和 CameraXVideoSpecResolver.kt 文件中，但这只是文件组织层面的调整，不需要改变模块边界或架构。

建议 2：SessionUiRenderModel 按 UI 区域拆分渲染模型

不建议落地

判定理由：已经按 UI 区域组织，拆分是伪需求

文件包含 23 个独立的 data class，每个对应一个 UI 区域的渲染模型。函数也已经按区域分组：

sessionSettingsRenderModel() → 设置面板
sessionSettingsPageRenderModel() → 设置页面
runtimeProControlsRenderModel() → 专业控制
filterLabPageRenderModel() → 滤镜实验室
watermarkLabSelectorRenderModel() → 水印选择器
portraitLabPageRenderModel() → 人像实验室
documentBatchOrganizerRenderModel() → 文档批处理
devLogRenderModel() → 开发日志
colorLabPanelRenderModel() → 色彩面板

这些函数都是 internal fun（顶层函数），不是类方法。它们接收 SessionState 和 AppTextResolver 作为参数，返回对应的 data class。这是 Kotlin 惯用的函数式风格，每个函数已经是独立的渲染逻辑单元。

拆分的唯一效果是把 23 个 data class 和 56 个函数分散到多个文件中。这不会减少任何复杂度，只是增加了文件数量。对于 Kotlin 项目，单文件组织同类的 data class 和工厂函数是常见且合理的做法。

建议 3：i18n 测试修复

应落地但范围远小于声明

实际状况：952 个测试中仅 5 个失败，其中确认 1 个是 i18n 问题。

SessionUiRenderModelTest.kt:1057
expected: "Temp Shift\nOff\nTap to cycle"
actual: "色温偏移\nOff\n..."

根因：测试使用 TestAppTextResolver 但该 resolver 在某些方法上返回了中文而非英文。测试环境的 Locale 默认为中文。

修复方案（二选一）：

  1. 在测试类上添加 @Config(qualifiers = "en") 或在 setUp 中设置 Locale.setDefault(Locale.US)
  2. 修复 TestAppTextResolver 使其对所有方法返回一致的英文测试字符串

注意：外部 Agent 声称"100 个测试失败，63% 是 i18n"是严重夸大的。实际仅 5 个失败，且 i18n 问题只是其中之一。

建议 4：app 模块编译错误排查

应落地 — 已定位根因

外部 Agent 声称的"Kotlin 编译器内部错误"完全错误。实际是两个普通的代码错误：

错误 1：linkEvents 参数缺失

MainActivity.kt:338 和 399 调用 devLogRenderModel() 时传递 linkEvents 参数
但 SessionUiRenderModel.kt:1695 的函数签名中该参数已被移除

修复：删除 MainActivity.kt 中两处 linkEvents = container.linkRecorder.snapshot(), 参数。

错误 2：when 表达式不完整

SessionUiRenderModel.kt:1625 defaultFilterLabFamily() 函数
when (activeMode) 缺少 ModeId.FULL_CLEAR 分支
ModeId.FULL_CLEAR 已在 core/mode 模块中定义

修复：在 when 表达式中添加 ModeId.FULL_CLEAR 分支，映射到合适的 FilterLabFamily（可能是 PHOTO）。

重要发现：单独运行 :app:compileDebugKotlin 时编译成功（使用缓存），但运行测试时触发重新编译才暴露错误。这说明错误是最近引入的，之前的工作区缓存掩盖了问题。

建议 5：DefaultCameraSession 复杂度降低

不建议落地

判定理由：这是 Session Kernel 核心，复杂度是职责的必然结果

DefaultCameraSession 是 AGENTS.md 定义的 Session Kernel，是整个应用的运行时核心。它负责：

  • Intent 分发（SessionIntent → 处理逻辑）
  • 生命周期管理（boot/shutdown/mode switch）
  • 预览状态机（starting/active/error/recovery）
  • 拍摄/录制协调
  • 缩放、亮度、对焦等运行时控制
  • 文档批处理状态管理

64 个方法处理这些职责是合理的。架构契约明确指出："Session Kernel is the only runtime owner for session state, preview/capture/recording state, recovery decisions, and state transitions." 这些职责集中在同一个类中是设计意图，不是技术债。

拆分的风险：将状态机逻辑分散到多个类会导致状态同步问题。当前的集中式设计保证了状态转换的原子性和一致性。拆分需要引入复杂的协调机制，得不偿失。

4. 诊断盲区与外部 Agent 分析缺陷

外部 Agent 分析的主要问题

  1. 指标迷信：仅凭圈复杂度（CC）和行数就建议拆分，没有分析代码的实际组织结构、架构契约、调用关系和职责边界。
  2. 数据失真：声称 100 个测试失败，实际仅 5 个。声称"Kotlin 编译器内部错误"，实际是普通的参数缺失和分支遗漏。
  3. 忽略架构契约：项目 AGENTS.md 明确定义了四层架构和各层职责边界。CameraXCaptureAdapter 是 Device Adapter 层、DefaultCameraSession 是 Session Kernel，这些都是架构设计意图，不是技术债。
  4. 混淆文件组织与模块边界：SessionUiRenderModel 的 23 个 data class 和 56 个函数已经是独立的逻辑单元，只是放在同一个文件中。文件组织 ≠ 模块架构。
  5. 未区分工具函数与类方法：CameraXCaptureAdapter 的 112 个方法中，大量是文件级 internal fun，不是类的成员方法。
