全局概览

codex_camera 是一个结构清晰的 Android 相机应用，采用 MVI/Elm 单向数据流架构，6 个 core 模块 + 7 个 mode 特性模块 + 1 个 app 层。核心设计决策（Session 状态机、Mode 插件化、Shot 生命周期模型）体现了良好的抽象意识。但随着功能增长，若干结构性债务正在积累：模式控制器 60-70% 的代码是结构性复制，5 个文件超过 1000 行，核心依赖图存在非自然边界。

以下矩阵将所有优化建议按 预期收益（纵轴）和 重构成本（横轴）定位。右上角是最佳 ROI 区域。

// core/mode 模块新增
class FrameRatioDelegate(private val context: ModeContext) {
    private val availableRatios = listOf(
        FrameRatio.RATIO_4_3, FrameRatio.RATIO_16_9, FrameRatio.RATIO_1_1
    )
    private var currentIndex = 0

    fun current() = availableRatios[currentIndex]
    fun cycle(): FrameRatio {
        currentIndex = (currentIndex + 1) % availableRatios.size
        return current()
    }
    fun select(ratio: FrameRatio): String? {
        val idx = availableRatios.indexOf(ratio)
        return if (idx >= 0) { currentIndex = idx; null }
        else "当前模式不支持 ${ratio.label} 画幅"
    }
}

class ProVariantDelegate(private val context: ModeContext) {
    var isEnabled = false; private set
    fun toggle(): Boolean { isEnabled = !isEnabled; return isEnabled }
    fun resolvedProfile(base: String) =
        if (isEnabled) "$base-pro" else base
}

S2: Session 上帝类拆分

高收益 高成本 中等风险 Sheaf / Local-to-Global

DefaultCameraSession.kt（1981 行）是整个系统中最大的行为类。它在一个 when 块中路由 ~30 种 Intent，同时管理生命周期、模式切换、预览绑定/恢复、拍摄流程、录制看门狗、设备图解析和状态更新。这是经典的 "God Object" — Grothendieck 会说：这些职责应该通过 "局部一致性条件" 自动组合成全局行为，而不是由一个中央协调者逐一调度。

// Session 变成薄壳 + 可组合的中间件链
class DefaultCameraSession(
    private val lifecycle: SessionLifecycleManager,
    private val preview: PreviewBindManager,
    private val capture: CaptureFlowManager,
    private val recording: RecordingManager,
    private val modeSwitch: ModeSwitchManager,
    private val graphResolver: DeviceGraphResolver,
) : CameraSession {

    override suspend fun dispatch(intent: SessionIntent) {
        // 中间件链：每个中间件处理自己关心的 Intent
        val handled = lifecycle.process(intent)
            || modeSwitch.process(intent)
            || capture.process(intent)
            || recording.process(intent)
            || preview.process(intent)
        // 未处理的 Intent 由默认逻辑处理
    }
}

// 提取预览绑定/恢复逻辑为独立 Manager
class PreviewBindManager(
    private val stateAccessor: () -> SessionState,
    private val emitEffect: suspend (SessionEffect) -> Unit,
    private val updateState: suspend (PartialState) -> Unit,
    private val trace: SessionTrace,
) {

        is SessionIntent.PreviewHostAttached -> { requestBinding(); true }
        is SessionIntent.PreviewHostDetached -> { handleDetach(); true }

    }
}

S3: 合同文件巨人症治理

高收益 低成本 高置信度 Ruthless Generalization

两个 "合同文件" 是项目中最大的文件：

  • SettingsContracts.kt: 1383 行 — 枚举、数据类、Action Reducer、序列化编解码器、默认滤镜配置、水印模板全在一个文件
  • MediaPipelineContracts.kt: 1134 行 — Shot 生命周期、ShotGraph 2.0、帧流合约、算法管道、媒体保存事务全在一个文件

Grothendieck 的 Ruthless Generalization 要求我们问：这些内容真的属于同一个抽象吗？ 答案是否定的 — 序列化逻辑不是合约，默认数据不是类型定义，Reducer 不是接口。它们被放在一起只是因为 "方便"，但这产生了隐性耦合：修改任何一部分都要重新编译整个文件的所有消费者。

S4: Effect-Device 依赖反转

中等收益 中等成本 Eliminate Artificial Boundaries

当前依赖图中，core/effect 直接依赖 core/device（通过 EffectCapabilityResolver 引用 DeviceCapabilities）。这创建了一条非自然的依赖路径：effect → device → media → settings，使得 effect 模块无法独立于设备层进行测试和演化。

Grothendieck 会问：如果这条边不存在，系统会更简单还是更混乱？ 答案是更简单 — effect 模块只关心 "这个 EffectSpec 是否被支持"，它不需要知道 DeviceCapabilities 的具体结构。

现状：effect 直接引用 device

S5: ShotExecutor / ShotGraph 统一

中等收益 中等成本 中等风险 Base Change

MediaPipelineContracts.kt 中并存两套管道模型：

  • ShotExecutor v1（line 497）：基于 CaptureStrategy → ShotPlan → ShotResult 的命令式管道
  • ShotGraph 2.0（line 592）：基于 CaptureNode → AlgorithmNode → OutputNode 的有向图模型

ShotPlan.toShotGraph() 扩展函数（line 893）将 v1 转换为 v2，说明 v2 是 v1 的超集。Grothendieck 的 Base Change 原则告诉我们：当同一结构在不同 "基" 上有两种表现时，应该通过一个统一的 Base Change 函子来连接，而不是保留两套平行代码。

现状：两套管道共存

// 转换桥接

目标：ShotGraph 为唯一管道
// 统一为 ShotGraph，ShotPlan 变成简化构造器
class UnifiedShotPipeline {
    fun plan(strategy: CaptureStrategy): ShotGraph {
        // 直接构建 ShotGraph，不再经过 ShotPlan 中间态
        return ShotGraphBuilder(strategy).build()
    }

    fun execute(graph: ShotGraph): ShotResult {
        // 按拓扑序执行节点
        val captures = executeCaptures(graph.captures)
        val processed = executeAlgorithms(graph.algorithms, captures)
        return assembleResult(graph.outputs, processed)
    }
}

S6: MainActivity 职责下沉

高收益 高成本 中等风险 Yoneda Perspective

MainActivity.kt（~1965 行）承担了远超 Activity 职责的工作：它既做 findViewById 绑定（~100+ 视图引用），又做状态渲染（render() 调用 7 个 render model），又做手势分发，又管理面板路由状态（activePanelRoute），又持有 latestSessionState 等 UI 状态。

Yoneda 引理告诉我们：一个对象完全由 "所有指向它的态射" 决定。 从调用者视角看，MainActivity 的 "态射"（被谁调用、传递什么数据）揭示了它实际上是 3 个不同对象的伪装：一个 ViewBinder、一个 StateRenderer、一个 PanelRouter。

现状：Activity 是万能容器

S7: render() 纯函数管道化

中等收益 低成本 Functoriality

当前 MainActivity.render() 方法顺序调用 7 个 render model 构建函数，然后逐一应用到视图。这本质上是一个 渲染管道：SessionState → [RenderModel1, RenderModel2, ...] → [ViewUpdate1, ViewUpdate2, ...]

Grothendieck 的 Functoriality 原则要求：所有操作都应该是函子 — 它们应该自动保持现有结构。 当前的 render() 不是函子的，因为它将 "数据转换" 和 "视图应用" 混合在一起，且依赖 Activity 的可变状态（activePanelRoute、latestSessionState 等）。

现状：混合转换和应用
private fun render(state: SessionState) {

    val controls = sessionControlsRenderModel(state, ...)
    val settings = sessionSettingsPageRenderModel(state, ...)
    val filter = filterLabPageRenderModel(state, ...)
    // ... 7 个 render model

    // 然后直接操作视图
    renderControlsSurface(controls)
    renderSettingsSurface(settings)
    renderFilterSurface(filter)
    // ... 每个 render 函数直接 findViewById 操作
}
目标：纯数据管道 + 单一应用点
