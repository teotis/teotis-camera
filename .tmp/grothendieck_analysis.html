2. 动机目录 (Motive Catalog)

以下是通过 Motive Discovery Protocol 发现的深层结构模式。每个动机代表一种"在不同位置以不同面貌出现、但在结构上完全同构"的深层架构元素。

动机 #1: 不可变状态归约器 Functor-like

定义：对不可变数据类执行 action → 返回新状态副本的纯函数模式。在代码库中出现 20+ 次。

成员位置：

  • PersistedSettings.reduce(PersistedSettingsAction) → PersistedSettings SettingsContracts.kt:971-1057
  • FeatureCatalog.reduce(FeatureCatalogAction) → FeatureCatalog SettingsContracts.kt:626-666
  • DefaultCameraSession.updateState() — 隐式归约 DefaultCameraSession.kt:1303-1357
  • 所有枚举的 fromStorageKey 查找 SettingsContracts.kt 多处

当前 vs 统一后对比：

当前

每个归约器手动实现 copy() + when 分支。SettingsContracts.kt 中 reduce 方法延伸 87 行，包含 26 个 when 分支。FeatureCatalog 的 reduce 有独立的 7 个分支。没有共享的错误处理或日志。

统一后

提取通用 Reducer 接口 + DSL。Settings 和 Catalog 的归约逻辑用同一套模式表达。自动获得日志、审计、undo/redo 能力。

动机 #2: 模式状态机 State-machine-like

定义：接收 ModeIntent → 更新内部循环状态 (cycle index) → 构建 ModeSnapshot → 返回 ModeSignal 的模式。在 7 个 ModeController 实现 中完全一致。

成员位置：

  • PhotoModeController feature/mode-photo/.../PhotoModePlugin.kt:54-408
  • NightModeController feature/mode-night/.../NightModePlugin.kt:48-486
  • PortraitModeController feature/mode-portrait/.../PortraitModePlugin.kt
  • VideoModeController, ProModeController, DocumentModeController, HumanisticModeController

结构签名：每个控制器都维护：

1. 一个或多个 cycle index (flashModeIndex, frameRatioIndex, profileIndex...)
2. buildSnapshot(headline, detail) 方法
3. onEnter/onExit/onSessionEvent 的几乎相同实现
4. deviceGraph() 构建相同的 DeviceGraphSpec.stillCapture/videoRecording
5. handle(ModeIntent) 中的 when 分发

当前 vs 统一后对比：

当前

7 个文件共 ~3,500 行代码，估算 60% 是结构重复。新增一个模式需要复制整个文件然后修改细节。修改共享行为需要改 7 个地方。

统一后

提取 ModeStateMachine 基类或组合。模式只需声明：有哪些可循环选项、每个选项的行为、如何构建策略。框架自动处理进入/退出/事件/快照。

动机 #3: 分层事件桥接 Functor-like

定义：在不同抽象层之间将事件/命令进行一对一的类型转换的模式。

成员位置：

  • SessionEffect → DeviceCommand CameraSessionCoordinator.kt:54-73
  • DeviceEvent → SessionIntent CameraSessionCoordinator.kt:76-113
  • SessionIntent → ModeIntent DefaultCameraSession.kt:138-141
  • ModeSessionEvent → ModeController.onSessionEvent

特征：每层桥接都是纯粹的 1:1 映射，但映射逻辑散落在不同类中，没有统一的"桥接注册表"，导致跨层追踪调用链需要跨越 4 个文件。

动机 #4: 后处理管道 Transform-like

定义：ShotResult → 依次经过多个 MediaPostProcessor 处理 → 富化的 ShotResult。

成员位置：

  • CompositeMediaPostProcessor MediaPipelineContracts.kt:317-327
  • PipelineMetadataPostProcessor MediaPipelineContracts.kt:329-380
  • MultiFrameMergePlaceholderPostProcessor MediaPipelineContracts.kt:382-411
  • App 层的 7 个 PostProcessor: DocumentAutoCrop, PhotoFrameRatio, PortraitRender, PhotoAlgorithm, PhotoWatermark, PhotoSelfieMirror, VideoWatermarkSubtitle

当前问题：PostProcessor 组合是手动的 (AppContainer 中硬编码列表)，无优先级/条件执行机制，无错误隔离。

动机 #5: 枚举存储编解码器 Set-like

定义：每个枚举类型都重复实现 storageKey + label + fromStorageKey 模式。

出现次数：20+ 个枚举，每个都有相同的 companion object { fun fromStorageKey(...) } 模板。

位置：SettingsContracts.kt 中集中了 12 个枚举，DeviceContracts.kt 中有 5 个，MediaPipelineContracts.kt 中有 4 个。

动机 #6: 元数据建造者 Functor-like

定义：在提交拍摄策略前，从多个源 (设置、运行时状态、模式) 聚合键值对构建 customTags Map 的模式。

成员位置：

  • PhotoModeController: 构建 20+ 对元数据 PhotoModePlugin.kt:139-168
  • NightModeController: 构建 15+ 对元数据 NightModePlugin.kt:183-203
  • ModeContext.captureAidMetadataTags() ModeContracts.kt:86-98
  • PipelineMetadataPostProcessor 再次解析元数据 MediaPipelineContracts.kt:329-380

反模式：元数据以弱类型 Map<String,String> 传递，生产者 (ModePlugin) 和消费者 (PostProcessor) 之间没有类型安全的契约。

发现 1: 七个"模式控制器"是同构投影

Difficulty⚙️ 中等 Benefit📈 高 Risk⚠️ 低 Confidence🎯 高
Motive Discovery Rising Sea Ruthless Generalization Grammar Invention

核心问题：PhotoModeController (408行), NightModeController (486行), PortraitModeController (631行), HumanisticModeController (522行) — 7 个 ModePlugin 实现共享完全相同的结构签名，但代码彼此完全独立。

Motive Discovery 视角：这 7 个类都是 模式状态机 动机的投影。每个都维护索引进位 (cycle index)、构建 ModeSnapshot、处理 ModeIntent。差异仅在于可循环选项的内容和拍摄策略的类型。

Rising Sea 视角：如果提升抽象层级，定义一个 ModeStateMachine 将"索引进位 → 构建快照 → 发射信号"作为通用框架，7 个实现缩减为声明式配置。每个模式只需声明：(1) 有哪些可循环选项组 (2) 每组选项如何影响策略 (3) UI 展示模板。

Grammar Invention 视角：需要发明的新概念：ModeOptionGroup — 一组可循环的互斥选项，关联到一个显示名称、一组值和选择行为。这个概念的引入让"闪光灯模式循环"、"画幅比例循环"、"场景风格循环"统一为同一个概念。

发现 2: 未命名的 "Reducer" 范式泛滥

Difficulty⚙️ 中等 Benefit📈 高 Risk⚠️ 低 Confidence🎯 高
Motive Discovery Grammar Invention Universal Property Functoriality

核心问题：代码库中有 20+ 个函数遵循 (State, Action) → State 签名，但没有统一的接口或命名。PersistedSettings.reduce(87行)、FeatureCatalog.reduce(40行)、DefaultCameraSession.updateState(55行) 各走各的实现路径。

Universal Property 视角：归约器的通用性质是：对于任意 Action 序列，reduce(reduce(state, a1), a2) ≡ reduce(state, combine(a1, a2))。目前每个归约器独立实现，没有共享的可组合性。

Functoriality 视角：归约操作应该是函子性的——将状态空间的变换保留到日志、审计、回滚等衍生空间。目前每次 reduce 调用手动调用 trace.record()，耦合了关注点。

Grammar Invention：需要正式承认 Reducer<S, A> 作为一等概念：typealias Reducer<S, A> = (S, A) -> S。这看似简单，但一旦命名，就可以建立 Reducer 组合子 (compose, combine, withLogging, withUndo) 的完整生态。

// 一等 Reducer 概念
fun interface Reducer<S, A> {
    fun S.reduce(action: A): S

    fun withLogging(log: (A, S, S) -> Unit): Reducer<S, A> = Reducer { state, action ->
        val next = state.reduce(action)
        log(action, state, next)
        next
    }
}

// 使用示例
val settingsReducer = Reducer<PersistedSettings, PersistedSettingsAction> { action ->
    when (action) { /* 原有逻辑 */ }
}.withLogging { action, prev, next ->
    trace.record("settings.reduced", "${action} -> ${next}")
}

发现 3: DefaultCameraSession 是"单块上帝对象"

Difficulty⚙️ 高 Benefit📈 高 Risk⚠️ 中 Confidence🎯 高
Rising Sea Sheaf Theory Artificial Boundaries Childlike Simplicity

核心问题：DefaultCameraSession.kt 包含 1,661 行代码，承担了：状态管理 (updateState 55行参数)、意图分发 (process 方法 166行 when 分支)、模式生命周期、倒计时管理、预览恢复协调、能力降级、镜头切换、变焦控制、分辨率切换、拍摄策略提交。所有关注点杂糅在一个类中。

Sheaf Theory 视角：一个健康的架构应当允许"局部一致的相邻段自动组合为全局"。但 DefaultCameraSession 中的 process() 方法是一个中心协调器，每个 handleXxx() 方法都在修改全局 _state。这是典型的 God Object 反模式，阻止了会话状态机的局部可组合性。

Rising Sea 视角：正确的抽象层级不是"一个处理所有意图的巨型类"，而是"一组可独立演化的状态机片段，各自处理一类意图，通过 sheaf 粘合条件自动组合"。

Childlike Simplicity 视角：新增一个 SessionIntent 变体需要在 process() 方法中添加新分支，在 updateState() 方法中添加新逻辑，修改大量现有代码。好的架构应该让"添加新意图"像"落下的果实"一样自然。

发现 4: PostProcessor 管道缺少基础设施

Difficulty⚙️ 低 Benefit📈 中 Risk⚠️ 低 Confidence🎯 高
Functoriality Sheaf Theory Yoneda Perspective

核心问题：AppContainer 中手动组合了 9 个 PostProcessor。如果一个处理器失败，后续处理器不会执行。没有条件执行（例如"仅在有人像深度数据时执行"），没有优先级排序，没有每个处理器的独立错误边界。

完整位置：AppContainer.kt:58-83 硬编码了处理器的实例化和顺序。

Functoriality 视角：PostProcessor 的组合应当是函子性的——一个处理器的输出自动成为下一个的输入，管道本身保持可组合性。当前 CompositeMediaPostProcessor 实现了基本组合，但缺少条件执行、错误隔离和优先级。

Yoneda 视角：不应从"我能实现什么 PostProcessor"出发设计接口，而应从"ShotResult 需要经过哪些后处理"出发。当前接口签名 process(ShotResult): ShotResult 过于通用，未体现不同处理器的不同输入需求。

发现 5: SettingsContracts.kt 是"数据类垃圾场"

发现 6: 能力降级逻辑分散

Difficulty⚙️ 中高 Benefit📈 高 Risk⚠️ 中 Confidence🎯 中高
Base Change Sheaf Theory Relative Point of View

核心问题：当 DeviceCapabilities 变化时（例如前置镜头不支持夜拍多帧），需要在一组分散的检查点手动处理降级：

  • NightModeController 检查 supportsNightMultiFrame 切换 profile 集合
  • DefaultCameraSession 检查 availableLensFacings 阻止镜头切换
  • DefaultCameraSession 检查 availableStillCaptureResolutionPresets 钳位分辨率
  • PhotoModeController 检查 supportsFlashControl 禁用闪光灯

每个检查点独立维护，没有全局的"能力→降级行为"映射。

Base Change 视角：设备能力变化本质上是一个 Base Change：从能力基线 A 切换到基线 B。Grothendieck 的方式是为每个 Base 定义一组"层上的层"，用 Base Change Functor 将结构从旧基线推送到新基线。当前的逐点检查相当于手动为每对 (代码点, 能力) 编写转换逻辑。

Relative Point of View 视角：每个功能点不应该问"当前能力是否支持 X？"，而应该声明"我需要能力 X"，由 Base Change Functor 自动判断当前能力集是否满足需求，以及不满足时应降级到什么行为。

发现 7: 元数据的弱类型链式依赖

Difficulty⚙️ 中等 Benefit📈 中 Risk⚠️ 中 Confidence🎯 中高
Yoneda Perspective Universal Property Motive Discovery

核心问题：拍摄元数据以 Map<String, String> 形式在 ModePlugin → ShotRequest → ShotResult → PostProcessor 之间传递。PostProcessor 通过硬编码字符串键读取值（如 "watermarkModel", "flash", "stillResolution"）。

Yoneda 视角：API 应该从消费者的使用模式反向推导。当前 PipelineMetadataPostProcessor 中 hardcode 了大量键名（如 customTags["selfieMirrorApply"]），这些键名是隐式契约——创建者与消费者之间的脆弱耦合。

Universal Property 视角：一个好的类型系统应该确保：如果模式插件声明它提供了某种元数据，后处理器可以安全假设它存在且类型正确。当前 Map<String,String> 既不是类型安全的，也不是文档完备的。

建议：定义 CaptureMetadata 数据类替代裸 Map。每个 ModePlugin 通过类型安全的建造者构建，每个 PostProcessor 通过类型安全的访问器读取。这一改动可以消除 PipelineMetadataPostProcessor 中 80% 的魔法字符串。

根因 1: 缺乏声明式状态管理框架

原始设计假设：采用 MVI (Model-View-Intent) 模式的 Intent/State/Effect 三元组就足以组织所有复杂度。

为什么这个假设失效了：MVI 提供了状态管理的 通信协议，但它不提供 组织原则。当 7 个模式、20+ 种意图、10+ 种状态字段汇聚到一个类时，MVI 只是确保通信格式一致，却不能阻止逻辑混乱。

下游影响：发现 1 (模式控制器重复)、发现 2 (Reducer 范式泛滥)、发现 3 (巨型 Session)

根因 2: 类型系统未充分表达领域概念

原始设计假设：Kotlin 的 data class + enum + sealed interface 已经足够表达相机应用的所有领域概念，无需自定义类型封装。

为什么这个假设失效了：当领域概念缺乏精确类型表达时，程序员使用弱类型容器 (Map<String,String>)、内联字符串解析和临时组合来弥补。这导致隐式契约、运行时错误和无法静态验证的一致性约束。

下游影响：发现 2 (Reducer 范式未命名)、发现 5 (Settings 垃圾场)、发现 7 (弱类型元数据)

根因 3: 模块边界按"功能领域"而非"变化速率"划分

原始设计假设：将代码按功能领域 (device, media, mode, session, settings) 划分模块自然产生内聚。

为什么这个假设失效了：功能边界可以产生 初始 内聚，但随着功能增长，同一功能领域内会出现以不同速率变化的子关注点。例如 SettingsContracts.kt 中的"枚举定义"极少变化，"滤镜配置"频繁变化，"序列化格式"中等变化——但它们在同一个文件/模块中以相同速率耦合。

下游影响：发现 5 (Settings 巨型文件)、发现 6 (能力降级分散)、发现 4 (PostProcessor 组合)

6. 研究方向 (Research Directions)

未解决问题

  • Real-time Preview Pipeline 的一致性条件：预览管道 (sensor→ISP→preview surface) 和拍摄管道 (sensor→post-processing→file) 之间是否应该共享后处理？哪些后处理必须在预览中实时反映（如滤镜），哪些只需在最终输出中处理（如水印）？
  • Live Photo 的动机归属：Live Photo 当前作为 CaptureStrategy.LivePhoto 实现。但从动机视角看，它究竟是"一种特殊的拍照策略"还是"一种新的媒体类型 (still+motion+sidecar)"？这个分类会影响后续 API 设计。
  • Pro Variant 到底是模式还是模式内的 Overlay：当前 Pro 按钮在各模式内部触发一个 variant。这是正确的抽象吗？还是 Pro 应当是一个能被注入到不同模式的 cross-cutting concern？

思想实验

  • 如果我们把 DeviceGraphSpec 提升为"Base"：当前 DeviceGraphSpec 在 ModeController 和 CameraSession 之间传递。如果将其显式参数化为拍摄的"Base"，并定义从 Base 到具体设备配置的 BaseChangeFunctor，当前分散在 DefaultCameraSession 中的 5 个 resolve* 方法将自动统一。
  • 如果我们把 ModeOptionGroup 作为一等类型：所有模式的"闪光灯循环"、"画幅循环"、"滤镜选择"其实是一个 CycleOption<T> 类型的不同实例。将这个概念提取为一等类型后，模式定义的代码量可以减少约 50%。

下一步调查

  • CameraXCaptureAdapter 未完全分析：该文件约 1,500 行，封装了 CameraX/Camera2 的复杂交互。本次分析只触及了其接口表面。它可能包含独立的架构问题（如 Android 生命周期处理、线程模型、Camera2 interop）。
  • 测试覆盖的架构影响：代码库中有 ~20 个测试文件。分析现有测试的覆盖范围可以揭示哪些架构决策被测试"锁定"，从而无法轻易重构。
  • 性能关键路径分析：首帧延迟 (firstFrameLatencyMillis)、连拍间隔、视频编码启动时间等性能指标与架构选择的关联值得深入调查。

7. 自评估 (Self-Assessment)

分析盲区

  • Android 生命周期和 Compose UI 层的架构分析未被包含，因为本次聚焦于 Core + Feature 模块。
  • CameraXCaptureAdapter 的完整实现 (约 1,500 行) 只被部分审查，其内部结构可能包含额外的重要发现。
  • 代码库的 git 历史未被分析，因此无法判断哪些文件是"修改热点"，哪些架构决策是最新的。

不确定性

  • ModeStateMachine 的声明式配置是否足够表达所有 7 个模式的差异？Night 模式的 Pro Variant、Pro 模式的手动参数草稿、Humanistic 模式的特殊滤镜体系等边缘情况需要实际验证。
  • BaseChangeFunctor 的集中化是否会产生新的瓶颈？当前分散的能力检查虽不优雅，但各检查点独立——集中后可能引入单点变更风险。
  • Reducer 接口化是否过度工程化？对于一个 28K 行的代码库，引入正式的 Reducer 接口可能看起来是"为模式而模式"。只有当它确实消除了大量重复代码时，才值得投入。

分析限制

本分析基于对代码的静态审查，未包含运行时行为观察、性能剖面分析或团队工作流的分析。所有"预计工作量"估计均基于代码结构的复杂度推断，实际工作量取决于团队对现有代码的熟悉程度和测试覆盖率。

分析总耗时：约 2 小时（代码阅读 + 分析 + 报告生成）。基于对 75 个 Kotlin 文件的全文阅读和对约 15 个核心文件的深度审查。
