OpenCamera 是一个采用分层模块化 MVI 架构的 Android 相机应用。项目整体架构质量较高：core 层定义纯 Kotlin 契约（无 Android 依赖），feature 层以插件模式实现相机模式，app 层作为组装点注入 Android 实现。

收益 × 重构成本 2×2 矩阵

横轴: 重构成本（人天估算） · 纵轴: 预期收益（1-10） · 点大小 = 置信度 · 红色 = 高风险

客观优势

  • 清晰的契约先行设计 — core/*/src/main/kotlin/.../Contracts.kt 系列文件在实现之前定义了完备的接口
  • 不可变状态管理 — 所有数据类使用 copy() + 函数式 reduce 模式，无副作用
  • MVI 单向数据流 — SessionIntent → SessionState → SessionEffect, 状态变化完全可追踪
  • 插件式模式扩展 — CameraModePlugin + ModeRegistry 实现了干净的模式发现与生命周期
  • 后处理管道 — CompositeMediaPostProcessor 支持声明式后处理链组合

二、11 项 Grothendieck 原则概要

1. 上升的海洋
拒绝局部补丁，提升抽象层次消解问题
2. 寻找绝对 Motive
收敛看似不同的多态表象到统一本质
3. 消除人为边界
好边界是事物的天然接缝，不是随意切割
4. 孩童般简洁
正确的基础让新功能像熟透的果实自然落下
5. 相对观点
不孤立分析 X，分析 X→S 的相对参数化结构
6. 泛性质优先
先定义契约和不变式，再证明实现存在
7. 函子性
好构造保持结构；所有变换应是函子
8. 层/局部到全局
局部一致性自动组合为全局对象
9. 无情泛化
不解决3个特例，找到包含它们的母问题
10. 基变换
跨环境的差异应统一为基变换函子
11. Yoneda 视角
对象由"所有指向它的态射"完全确定

发现 1: PostProcessor 管道硬编码注册

难度: 中低 收益: 7/10 风险: 低 置信度: 高 原则 1 · 上升的海洋 原则 8 · 层/局部到全局

问题描述

app/.../AppContainer.kt:58-84 — 在 AppContainer 的构造函数中，mediaPostProcessor 是一个手工维护的列表，包含 9 个后处理器。每新增一个后处理器，都必须修改 AppContainer。这是一种"局部补丁"式设计——问题的水位线太低。

Grothendieck 会问：为什么需要手动编排这 9 个处理器？它们之间的组合关系是否应该是处理器自身的属性？

interface MediaPostProcessorPlugin {
    val priority: Int  // 越小越先执行
    fun isApplicable(result: ShotResult): Boolean
    fun create(context: Context): MediaPostProcessor
}

// AppContainer 变为
class AppContainer(context: Context) {
    private val processorPlugins = listOf(
        MultiFrameMergePlugin(),
        DocumentAutoCropPlugin(),
        // ... 各插件通过 ServiceLoader 或显式列表

发现 2: Mode 插件中的重复模式 — 未被发现的 Motive

难度: 中高 收益: 8/10 风险: 中 置信度: 高 原则 2 · 绝对 Motive 原则 9 · 无情泛化

问题描述

观察 feature/mode-photo/.../PhotoModePlugin.kt 和 feature/mode-pro/.../ProModePlugin.kt，它们共享一种深层结构：

  • buildSnapshot(headline, detail) — 构造 UI 快照

发现 3: 渲染模型层膨胀 — 人为边界造成的翻译税

难度: 中高 收益: 6/10 风险: 中高 置信度: 中高 原则 3 · 消除人为边界 原则 4 · 孩童般简洁

问题描述

app/.../SessionUiRenderModel.kt 长达 2244 行，定义了 20+ 个 RenderModel 数据类和大量转换函数。核心问题：在 SessionState（数据）和 UI 渲染之间人为插入了一层"RenderModel"，但这层并没有真正简化问题——它只是把复杂性从 View 层"移动到了别处"。

关键指标：sessionSettingsPageRenderModel() 函数约 300 行，构造一个包含 6 层嵌套数据类的渲染模型。每个 SettingsControlRenderModel 需要 5-7 个参数，其中大部分是从 SessionState 计算的衍生值。

这不是真正的抽象——这是 Grothendieck 所说的虚假泛化。特殊性代码没有消失，它只是从 MainActivity.render() 移动到了 SessionUiRenderModel.kt 中的巨型函数。

发现 4: 设置系统的全有或全无重建

难度: 中 收益: 7/10 风险: 中 置信度: 高 原则 5 · 相对观点 原则 10 · 基变换

问题描述

core/settings/.../SettingsContracts.kt:930-969 — PersistedSettingsAction 是一个包含 19 个变体的 sealed interface。每次添加新设置需要：

  1. 在 PersistedSettings 中添加字段
  2. 在 PersistedSettingsAction 中添加变体
  3. 在 PersistedSettings.reduce() 中添加分支
  4. 在 SessionSettingsPageRenderModel 构建函数中添加渲染逻辑
  5. 在 MainActivity 中添加按钮绑定

这不是"孩童般简洁"——这是繁琐的仪式性编码。更关键的是，当 DefaultCameraSession.kt:295-354 中 handleSettingsUpdated 完全销毁并重建 ModeController 时（通过 currentController.onExit() + createController() + onEnter()），说明设置变更与模式控制器的关系未被正确模型化。

Grothendieck 的相对观点：设置变更应该表示为"基变换"（base change），ModeController 应该是基空间上的一个族（family over the base），基变换自动拉回（pullback）到新模式配置，而不是重建整个对象。
Grothendieck 式问题：如果设置是 ModeController 的"基"，那么修改一个设置应该是基空间上的一个态射，ModeController 应该函子性地响应这个态射——而不是重建。

具体方案

将设置建模为"差异更新"而非"全量快照"：

展开代码草图 
// 不是:
session.dispatch(SessionIntent.SettingsUpdated(fullSnapshot))
currentController.onExit()
currentController = createController(...) // 重建

// 而是:
sealed interface SettingsDelta {
    data class PhotoFilterChanged(val id: String) : SettingsDelta
    data class VideoSpecChanged(val spec: VideoSpec) : SettingsDelta
    // ...
}
// ModeController 接收增量更新
interface ModeController {
    suspend fun onSettingsDelta(delta: SettingsDelta) = Unit
}
// Session 转发 delta 而不是重建 controller
session.dispatch(SessionIntent.SettingsDelta(delta))
currentController.onSettingsDelta(delta) // 就地更新

发现 5: ModeContext 通过闭包捕获可变状态 — 违反引用透明性

难度: 中 收益: 5/10 风险: 低 置信度: 高 原则 5 · 相对观点 原则 7 · 函子性

问题描述

core/mode/.../ModeContracts.kt:63-80 — ModeContext.runtimeState 是一个 lambda：val runtimeState: () -> ModeRuntimeState，它在 core/session/.../DefaultCameraSession.kt:1514-1520 中被定义为捕获了 sessionDeviceCapabilities、sessionLensFacing 等可变变量的闭包。

这意味着 ModeController 调用 context.runtimeState() 时，得到的结果依赖于 Session 中可变字段的当前值——而不是一个明确的、参数化的输入。这是引用不透明（referentially opaque）的，使得 ModeController 的单元测试必须依赖一个完整的 Session 状态环境。

发现 6: FilterLabFamily 的分支逻辑重复

难度: 低 收益: 5/10 风险: 低 置信度: 很高 原则 9 · 无情泛化 原则 7 · 函子性

问题描述

类似地，SessionSettingsManager.kt:213-225 的 familyDefaultAction() 又是一个 4 路 when 分支做同一件事。

发现 7: DeviceCapabilities 散落的守卫检查 — 缺少基变换函子

难度: 高 收益: 8/10 风险: 中高 置信度: 中 原则 10 · 基变换 原则 1 · 上升的海洋

问题描述

在整个代码库中，DeviceCapabilities 的检查散落在各处：

  • SessionUiRenderModel.kt — supportsStillCapture 检查出现 20+ 次
  • PhotoModePlugin.kt — supportsFlashControl 检查
  • DefaultCameraSession.kt — availableLensFacings.size < 2 检查
  • 每个 ModePlugin 的 isSupported() 是在插件层的守卫

Grothendieck 的基变换理论：同一结构在不同基上的行为变异应由基变换函子统一，而非 N 份发散代码。

Grothendieck 式思考：如果 DeviceCapabilities 是"基空间"，那么整个相机管道应该是基空间上的一个族（family）。更换设备（改变基）应该通过一个统一的 baseChange(capabilities) 函子来拉回整个管道配置，而不是在每个使用点检查 if (supportsXxx)。

发现 8: ModeController 接口的 Yoneda 审视 — 从实现者视角设计

问题描述

core/mode/.../ModeContracts.kt:155-180 — ModeController 接口定义了 9 个方法，但多数实现只是 no-op：

Yoneda 引理告诉我们：一个对象完全由"指向它的所有态射"——即所有使用它的方式——决定。反向从调用者视角看：Session 调用这些方法的模式是什么？每次状态变更时全部调用一遍。这意味着接口应该反映"调用模式"而非"实现者的能力列表"。
