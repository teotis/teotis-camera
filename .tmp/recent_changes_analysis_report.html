StillCaptureGraphHelper — 接口分析

fun stillCaptureDeviceGraph(runtimeState: ModeRuntimeState): DeviceGraphSpec {
    return DeviceGraphSpec.stillCapture(
        preferredLensFacing = runtimeState.lensFacing,
        enablePreviewSnapshots = runtimeState.deviceCapabilities.supportsPreviewSnapshots,
        qualityPreference = runtimeState.stillCaptureQuality,
        resolutionPreset = runtimeState.stillCaptureResolutionPreset
    )
}

这是一个薄封装层：将 ModeRuntimeState 的属性映射为 DeviceGraphSpec.stillCapture() 参数。消除了 6 个控制器中的重复参数映射代码。

StillShotSessionEventReducer — 消除重复

fun reduceStillShotSessionEvent(
    event: ModeSessionEvent,
    text: StillShotSessionEventText,
    updateSnapshot: (headline: String, detail: String?) -> Unit
): Boolean {
    return when (event) {
        is ModeSessionEvent.ShotStarted -> { /* video filter → false */ }
        is ModeSessionEvent.ShotCompleted -> { /* video filter → false */ }
        is ModeSessionEvent.ShotFailed -> { /* video filter → false */ }
    }
}

所有 Still 模式的 ShotStarted/ShotCompleted/ShotFailed 事件处理被集中到此函数。video 事件通过 mediaType 检查被过滤（返回 false），still 事件则调用 updateSnapshot 更新 UI。

ProVariantState — 共享状态设计

ProVariantState 封装了 Pro 模式的标准/Pro 变体切换逻辑：

  • toggle() — 切换 isEnabled 并返回 ProVariantToggleResult（含 hint 文字）
  • manualControlsEnabled() — 查询设备是否支持手动控制
  • resolvedAlgorithmProfile(base) — 根据 pro/manual 状态追加后缀（-pro, -pro-assist）
  • metadataTags() — 生成 EXIF 元数据标签
  • variantExifLabel() — "Pro" vs "Pro Assist"

该类不持有协程/事件sink引用，通过 ModeContext 间接访问运行时状态，保持纯逻辑层的可测试性。

高 修复前的缺陷: 节点引用不一致

在 LIVE_PHOTO 和 VIDEO_RECORDING 分支中，filter/watermark/thumbnail 算法的 input 节点之前硬编码为 "shot:primary"，但实际情况是：

  • LIVE_PHOTO: primary capture node 的 id 是 "shot:still"，不是 "shot:primary"
  • VIDEO_RECORDING: primary capture node 的 id 是 "shot:video"，不是 "shot:primary"

影响: Live Photo 和视频录制时，滤镜/水印/缩略图算法引用了不存在的 capture node，可能导致算法跳过或失败。

修复: 引入 primaryCaptureNodeId 变量，在每个 when 分支中赋值为正确的 node id，然后 algorithm 节点统一引用该变量。

resolePhotoOutputHandle 兼容性修复

参数类型从 Uri? 恢复为 String?。因为在 JVM 单元测试环境中 Uri.parse() 可能返回 null（测试没有 Android framework），导致下游逻辑拿不到有效的 output handle。Uri → String 转换移至调用点（CameraXCaptureAdapter），由 Android 环境中可靠的 Uri.parse 执行。

七、潜在问题排查清单

发现问题总计: 8 个 (高 1 | 中 3 | 低 3 | 观察 1)

P1 lightPaletteBaseSpec 状态清理不一致

位置: MainActivity.kt:380-382 & CockpitPanelRouter.kt:140-147

问题: lightPaletteBaseSpec 仅在 onBackPressed() 中通过检测前一个路由是否为 StyleLab/ColorLab 来清理。但通过 X 按钮关闭（CloseFilterLab 命令）或 DismissAll 命令关闭时，lightPaletteBaseSpec 不会被清理。

场景:

  1. 用户打开滤镜面板 → 进入 Light 调色模式 → lightPaletteBaseSpec 被设置
  2. 用户点击 X 按钮关闭 → CloseFilterLab 命令 → 路由重置但 baseSpec 未清理
  3. 用户更改滤镜设置 → 重新打开滤镜 → lightPaletteBaseSpec 仍保留旧值
  4. 用户触摸调色板 → handleFilterPaletteTouch() 使用过时的 baseSpec 计算

风险: 调色板计算结果基于错误的基准 FilterRenderSpec，可能导致颜色/色调偏移量计算错误。

建议修复:

// 在 onBackPressed() 中或每次面板关闭时
if (activePanelRoute is CockpitPanelRoute.None) {
    lightPaletteBaseSpec = null
}

P2 render() 方法中不更新 lightPaletteBaseSpec

位置: MainActivity.kt:203-286 (render 方法)

问题: render() 直接调用 filterLabRenderer.renderPage(filterLabPage) 而不检查/更新 lightPaletteBaseSpec。baseSpec 的更新逻辑仅在 renderLatestFilterLab() 中（line 498-500）。当 session state 更新触发 render() 时，如果 filter adjustment panel 可见，baseSpec 不会自动同步。

风险: 如果 session state 在 filter panel 打开期间更新（例如其他设置变化），lightPaletteBaseSpec 可能落后于当前 renderSpec。

P3 humanistic 模式入口被硬编码禁用

位置: MainActivityActionBinder.kt:291-292

views.modeTrack.humanistic.visibility = View.GONE
views.modeTrack.humanistic.setOnClickListener(null)

问题: Humanistic 模式在 mode track 中被无条件隐藏，且点击监听被设为 null。但 HumanisticModePlugin 仍在 ModeRegistry 中注册（AppContainer.kt:89），其控制器逻辑仍存在于 feature/mode-humanistic 模块。

风险: 这是否为临时禁用还是永久决定？如果是临时禁用，应该通过 Feature Flag 控制而非硬编码；如果是永久移除，应该清理相关模块和注册。

建议: 确认意图。如果是功能门控，应使用可配置的 feature flag。

P4 StillCaptureGraphHelper 抽象层级过薄

位置: core/mode/StillCaptureGraphHelper.kt

问题: 该函数仅做参数转发，没有增加任何逻辑或抽象价值：

fun stillCaptureDeviceGraph(runtimeState: ModeRuntimeState): DeviceGraphSpec {
    return DeviceGraphSpec.stillCapture(
        preferredLensFacing = runtimeState.lensFacing,
        enablePreviewSnapshots = runtimeState.deviceCapabilities.supportsPreviewSnapshots,
        qualityPreference = runtimeState.stillCaptureQuality,
        resolutionPreset = runtimeState.stillCaptureResolutionPreset
    )
}

风险: 薄封装层增加了间接性但未提供新的抽象价值。如果后续需要添加逻辑（如条件化参数映射），这里是一个合理的扩展点，但目前它只是重命名参数的适配器。

建议: 考虑合并到 ModeContext 的扩展函数中，或者在需要条件化逻辑时再引入。

P5 SessionSettingsManager TOCTOU 潜在竞态

位置: SessionSettingsManager.kt:49-68

问题: apply() 方法先检查 session.state.value.activeShot != null，然后执行 save/dispatch。虽然协程在主线程调度降低了竞态风险，但 state.value 是 snapshot read——在检查点和实际操作点之间，状态可能已经变化。

实际风险: 低。因为所有操作都在主线程的协程上下文中执行，且 CameraSession 的状态更新也发生在同一调度上下文中。但如果未来引入异步状态更新，此处的 TOCTOU 窗口会扩大。

P6 gestureRouter 非空断言 (!!)

位置: MainActivityActionBinder.kt:357

previewView.setOnTouchListener { v, event ->
    gestureRouter!!.onTouchEvent(v, event)
}

问题: 如果 setOnTouchListener 在 bind() 完成前被触发（例如 Fragment 重建场景），gestureRouter 可能为 null。

实际风险: 低。bind() 在 onCreate 中同步调用，view 在此时还未对用户可见。但使用 !! 是一个 code smell，建议改为 lateinit 配合 checkNotNull 或 let。

P7 onBackPressed 中同时检查 styleLab 和 colorLab 的 lightPaletteBaseSpec 清理

位置: MainActivity.kt:380-382

if (previousRoute is CockpitPanelRoute.StyleLab || previousRoute is CockpitPanelRoute.ColorLab) {
    lightPaletteBaseSpec = null
}

问题: 该检查使用 || 同时匹配 StyleLab 和 ColorLab，但如果 panel 路由在 back 之前已经通过 reducer 改变了（例如 onBackPressed 中调用了 panelRouter.reduce(CockpitPanelCommand.AndroidBack) 行375），previousRoute 捕获的是 reduce 之前的 route。这个逻辑正确，但依赖 previousRoute 的局部捕获，如果后续有人重构移动了 reduce 调用的位置，可能导致检查的 route 不对。

建议: 将 lightPaletteBaseSpec 的清理逻辑下放到 CockpitPanelRouter 的 AndroidBack 处理中，或在 MainActivityRenderer.renderPanelVisibility 中统一处理。

观察 CapabilityContracts 删除后引用完整性

位置: core/device/CapabilityContracts.kt (已删除)

验证: 删除前 Contract 已迁移至 core:capability 模块。AppContainer.kt 中通过 asCapabilityGraphQuery() 扩展函数引用新模块。所有引用点应已通过编译验证。如果 CI 编译通过，则引用完整性无问题。

关注点: 是否有其他模块通过传递依赖间接引用了旧路径的类型？Gradle 编译会捕获直接引用问题，但如果有模块声明了 compileOnly 或 runtimeOnly 依赖，可能需要额外验证。

lightPaletteBaseSpec 状态清理一致性问题 — 这是唯一的高优先级问题。修复方式简单：在 onBackPressed 方法中，当 route 变为 None 时清理 baseSpec，覆盖 X 按钮关闭和 back 键两种路径。

架构评价

  • MainActivity 瘦身: 是一次成功的重构，将单一 God Object 拆分为 Views/Renderer/ActionBinder/Callbacks 四个职责清晰的组件。行数从 1820 降到 711（-61%），每个新组件都在 30-360 行范围内，可测试性大幅提升。
