# OpenCamera 2.0 准入：高难10%与多模态限定任务（主控方案）

> 该文档由“多模态/高阶主控 agent”独占执行；非多模态 agent 不承接。

## 1. 任务边界

本任务只做最难且跨域耦合最强的部分：

1. 统一 2.0 判据（设计语义、交互语法、状态机、产物链路）
2. 多模态视觉审查（界面逻辑一致性、状态可感知性）
3. 端到端证据拼接（输入动作 → session/device/media → 输出可验证）
4. 决策级结论（GO / CONDITIONAL GO / NO GO）

---

## 2. 设计执行方案

### 2.1 建立“2.0 准入判据矩阵”（必须先做）

输出一个 `Readiness Matrix`，包含：

- 维度：UI 逻辑 / 交互流畅 / 功能可用 / IO 通畅
- 子项：每维至少 6 个可观测子项
- 证据：代码位点 + 测试位点 + 视觉位点 + 结果位点
- 判定：Pass / Risk / Fail
- 风险等级：P0/P1/P2/P3

### 2.2 多模态视觉对齐审查

对照以下来源做一致性仲裁：

- 设计与规范：`codex/v2_ui/*.md`
- 视觉参考：`codex/v2_ui/reference_images/*`
- 近期反馈计划：`codex/agent_plans/2026-05-22-*multimodal*`、`*feedback*`
- 实现与渲染模型：`app/.../camera`、`SessionUiRenderModel*`

重点检查：

- 模式轨道可读性与可触达性
- 快门区与次级面板视觉层级
- 横竖屏布局一致性
- 状态提示（录制/夜景/降级/恢复）表达统一

### 2.3 IO 链路端到端归因

以“高频主路径”构建四条金路径：

1. 标准拍照
2. 录像开始/停止
3. 模式切换后拍照
4. 权限恢复后再次拍照

每条路径必须形成：

- 输入动作（UI Intent）
- Session 状态迁移（state/effect）
- Device 请求翻译（adapter/request）
- Media 产物（缩略图/文件/元数据）
- 用户可见反馈（UI 状态）

### 2.4 评审与结论输出

形成决策稿：

- `GO` 条件满足情况
- 未满足项的风险性质
- 若 `CONDITIONAL GO`：列出 2 周内必须清零项

---

## 3. 难点与破解策略

### 难点 A：跨文档版本漂移

- 现象：设计文档、代码实现、测试断言可能不在同一语义版本。
- 策略：以“用户可见行为”为第一真值，其他证据做佐证，避免文档绑架结论。

### 难点 B：视觉问题难以量化

- 现象：UI 讨论容易主观化。
- 策略：把“可读、可点、可理解”拆成客观项（字号/对比/命中区/状态显隐），统一量化表。

### 难点 C：本地难复现真实设备问题

- 现象：provider death、长稳恢复等依赖真实设备。
- 策略：区分“可本地证伪”与“需真机确认”项，输出显式风险标签，不混为已通过。

### 难点 D：架构正确但体验断裂

- 现象：内核链路正确，但用户感知失败（慢、无反馈、入口深）。
- 策略：每个架构结论都要求至少一个用户可见证据与一个结果证据。

---

## 4. 输出模板（主控专用）

1. `Readiness Matrix`（总表）
2. `Top Risks`（仅 P0/P1）
3. `Decision`（GO/CONDITIONAL/NO GO）
4. `2-Week Closure Plan`（若非 GO）
