# OpenCamera 2.0 准入综合审查总控方案（Master Plan）

## 0. 背景与目标

本方案用于审查项目是否可“整体进入 2.0 标准”，评估口径聚焦四项：

1. **UI 设计逻辑自洽**
2. **用户交互流畅**
3. **可触及功能有效可用**
4. **输入输出链路通畅**

并在仓内现实约束下（当前 Stage 7，约 80% 进度）输出可落地结论：

- `GO`：可整体进入 2.0（允许少量非阻断缺陷）
- `CONDITIONAL GO`：满足核心闭环，但存在需在短周期清零的问题
- `NO GO`：存在关键链路断裂或高风险缺陷

---

## 1. 用户需求总结（基于历史会话意图）

从仓内 `codex/plan.md`、`codex/prompt.md`、`codex/documentation.md` 与既有 agent 计划可抽象出稳定用户诉求：

- 不是“功能点堆叠”，而是**端到端体验完整性**。
- 不是“单测通过即完成”，而是**真实用户可感知链路稳定**。
- UI 改造目标不是视觉独立优化，而是**交互语法、信息架构、模式切换、预览与输出反馈的一致性**。
- 对 Stage 7 的期望是把“可观测、可恢复、可验证”形成治理闭环，避免隐式故障沉默。

简言之：用户要的是“能交付、可验证、可持续迭代”的相机 2.0 基线，而不是 demo 完成度。

---

## 2. 对标友商视角（审查维度，不做品牌复刻）

以主流系统相机（Apple/Google/小米/OPPO/vivo/Huawei 一线产品）共同特征为对标抽象：

- **入口可预期**：模式、快门、缩略图、设置入口认知负担低。
- **状态可理解**：录像/夜景/HDR/闪光/防抖/手动参数状态可见且一致。
- **动作可恢复**：权限中断、前后台切换、设备温控、预览首帧异常有明确反馈与恢复路径。
- **结果可确认**：拍后缩略图及时、媒体可打开、元数据与视觉结果一致。
- **性能可接受**：首帧、快门响应、模式切换、面板开合延迟在主观可接受阈值内。

本项目评估采用“对标能力框架”，不要求像素级 UI 一致，而要求体验语义一致。

---

## 3. 项目定位判断（当前仓内）

当前 `OpenCamera` 定位可定义为：

- **架构优先的 Android 相机内核工程**（Session Kernel / Device Adapter / Media Pipeline / Mode Plugin 清晰分层）
- 在 Stage 7 已形成一定可观测与恢复治理基础（含 preview stall watchdog、thermal issue forwarding 等）
- UI/交互正从功能可用走向 2.0 的“系统化体验”阶段（见 `codex/v2_ui/*` 与近期 agent plans）

因此，2.0 准入评估必须是“架构稳定性 + UX 闭环”双轴，不可偏废。

---

## 4. 审查总体方法

### 4.1 四层证据法

每个结论必须至少来自以下两类证据：

- **静态证据**：代码、契约、状态机、用例、文档
- **动态证据**：单测/集成测试/构建验证/脚本验证
- **视觉证据**：界面截图、状态覆盖、交互路径录屏或序列图
- **结果证据**：媒体文件、元数据、错误日志、恢复日志

### 4.2 风险分级

- `P0` 阻断：核心链路不可用/高概率崩溃/结果不可信
- `P1` 关键：核心链路可用但体验明显断裂
- `P2` 重要：可用但不顺滑或一致性不足
- `P3` 优化：可后续迭代

### 4.3 最终判定门槛（2.0）

- `P0 = 0`
- 高优先用户主链路（拍照/录像/模式切换/结果回看/权限恢复）`P1` 总量低于阈值
- Stage 7 相关观测与恢复路径“有证据可回放”

### 5.1 多模态主控（本轮由你指定我承担）

- C 组：功能可触达与有效性回归
- D 组：IO 链路与媒体结果核验
- E 组：稳定性、恢复与观测基线复验
- F 组：发布准入清单与风险台账汇总

## 8. 关键难点分析（总览）

1. **“看起来可用”与“可稳定交付”之间的证据鸿沟**
2. **UI 文档、实现、测试三者版本不同步导致的假阳性**
3. **多模式与多状态叠加下交互语法冲突（尤其横竖屏与面板并发）**
4. **真实设备相关问题在本地环境难以完全复现**
5. **性能/时延阈值缺少设备分层基线，易引发争议**
