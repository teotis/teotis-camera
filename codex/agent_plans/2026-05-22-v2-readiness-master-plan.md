# OpenCamera 2.0 准入综合审查总控方案

> 用途：本文件是总控分发入口。直接把第 7 节对应文档发给不同非多模态 agent；多模态/高难主控只执行 `hard10` 文档，不和其他 agent 抢普通静态检查。

## 0. 背景与目标

本方案用于审查项目是否可“整体进入 2.0 标准”，评估口径聚焦四项：

1. **UI 设计逻辑自洽**
2. **用户交互流畅**
3. **可触及功能有效可用**
4. **输入输出链路通畅**

并在仓内现实约束下（当前 Stage 7，约 80% 进度）输出可落地结论：

- `GO`：可整体进入 2.0，允许少量非阻断缺陷。
- `CONDITIONAL GO`：满足核心闭环，但存在需在短周期清零的问题。
- `NO GO`：存在关键链路断裂或高风险缺陷。

注意：本审查不授权进入新 stage，不替代 Stage 7 exit checklist，也不允许绕开既有四层主链路新增隐藏内核。

## 1. 用户需求总结

从 `codex/plan.md`、`codex/prompt.md`、`codex/documentation.md`、`codex/v2_ui/*`、`codex/capability_kernel_v2/*` 与近期真机反馈方案可抽象出稳定用户诉求：

- 不是“功能点堆叠”，而是**端到端体验完整性**。
- 不是“单测通过即完成”，而是**真实用户可感知链路稳定**。
- UI 改造目标不是视觉独立优化，而是**交互语法、信息架构、模式切换、预览与输出反馈的一致性**。
- 对 Stage 7 的期望是把“可观测、可恢复、可验证”形成治理闭环，避免隐式故障沉默。
- 对 2.0 的期望不是追求厂商系统相机的私有 ISP/算法同等能力，而是把公开 Android 能力、App 侧处理、诚实降级和产品体验组织成一条可信链路。
- 历史真机反馈的核心不是“再加几个按钮”，而是解决缩略图跳变、横屏几何、画幅、面板信息架构、中文可读性、色彩实验室入口、模式栏命中、保存输出可信度等用户实际可感知断点。

简言之：用户要的是“能交付、可验证、可持续迭代”的相机 2.0 基线，而不是 demo 完成度。

## 2. 对标友商视角

以主流系统相机（Apple/Google/小米/OPPO/vivo/Huawei 一线产品）共同特征为对标抽象；当需要具体产品表达参考时，按项目规则优先参考 `vivo X300 Ultra`，次优参考 `iPhone 17 Pro Max`，但只吸收产品语义，不做品牌复刻。

- **入口可预期**：模式、快门、缩略图、设置入口认知负担低。
- **状态可理解**：录像/夜景/HDR/闪光/防抖/手动参数状态可见且一致。
- **动作可恢复**：权限中断、前后台切换、设备温控、预览首帧异常有明确反馈与恢复路径。
- **结果可确认**：拍后缩略图及时、媒体可打开、元数据与视觉结果一致。
- **性能可接受**：首帧、快门响应、模式切换、面板开合延迟在主观可接受阈值内。
- **功能分层清晰**：一线高频动作留在取景器 cockpit；中频配置进入快捷/风格/色彩实验室；低频或高风险项进入设置/Dev/诊断。
- **降级不撒谎**：不支持或受限能力不伪装成已执行，必须呈现 `unsupported` 或 `degraded` 的产品语义。

本项目评估采用“对标能力框架”，不要求像素级 UI 一致，而要求体验语义一致。

## 3. 项目定位判断

当前 `OpenCamera` 定位可定义为：

- **架构优先的 Android 相机内核工程**：`Session Kernel / Device Adapter / Media Pipeline / Mode Plugin` 清晰分层。
- **通用 Android 相机 App**：不依赖厂商系统相机权限，不承诺 native ISP/BSP 私有链路能力。
- **2.0 cockpit 体验工程**：UI/交互从功能可用走向系统化体验，见 `codex/v2_ui/*`。
- **能力治理工程**：能力图谱、算法管线、资源预算、Live/时序媒体等进入 `codex/capability_kernel_v2/*` 设计包。

`OpenCamera 2.0` 应定位为：一个以清晰相机会话内核为底座、面向通用 Android 设备、用诚实能力治理和一致 cockpit 体验承载复杂拍摄功能的专业相机 App。

### 3.1 当前不是 2.0 的风险信号

以下任一信号若被证实为 P0/P1，就不能直接判 `GO`：

- 用户能看到入口，但入口只改 UI 状态，不影响 session/device/media 请求。
- 拍照或录像完成后，缩略图、文件、元数据、后处理结果之间互相打架。
- 横竖屏、画幅、网格、tap 区域使用不同几何来源。
- 面板路由存在多个并发 owner，导致返回、外点关闭、状态显示不一致。
- 预览恢复、权限恢复、provider stall 等 Stage 7 问题只在日志中存在，用户无反馈或无法复现诊断。

## 4. 审查总体方法

### 4.1 四层证据法

每个结论必须至少来自以下两类证据：

- **静态证据**：代码、契约、状态机、用例、文档。
- **动态证据**：单测、集成测试、构建验证、脚本验证。
- **视觉证据**：界面截图、状态覆盖、交互路径录屏或序列图。
- **结果证据**：媒体文件、元数据、错误日志、恢复日志。

非多模态 agent 只能产出静态、动态、文本日志和文本序列图证据；截图、录屏、保存图视觉判断必须交给主控或多模态 QA。

### 4.2 风险分级

- `P0` 阻断：核心链路不可用、高概率崩溃、结果不可信。
- `P1` 关键：核心链路可用但体验明显断裂。
- `P2` 重要：可用但不顺滑或一致性不足。
- `P3` 优化：可后续迭代。

### 4.3 最终判定门槛

- `GO`：`P0=0`，主链路 `P1<=2`，且每个 `P1` 都有明确 owner 和不影响主流程的规避方式。
- `CONDITIONAL GO`：`P0=0`，主链路 `P1<=6`，且 2 周内有可执行清零计划。
- `NO GO`：存在任一 P0，或拍照/录像/保存/权限恢复任一金路径不可证明通畅。

### 4.4 2.0 Readiness Matrix

| 维度 | 子项 | Pass 证据 | Risk/Fail 信号 | 主责文档 |
| --- | --- | --- | --- | --- |
| UI 逻辑自洽 | 顶部栏、侧边栏、底部 cockpit 分层 | `CameraCockpitRenderModel`、`CockpitPanelRoute`、XML 结构与 `v2_ui` 一致 | 同一入口出现在多个层级，或风格/色彩/设置/Dev 归属混乱 | A |
| UI 逻辑自洽 | 文案、i18n、窄屏可读 | string resources、`AppTextResolver`、render model 测试 | 用户可见英文残留、中文省略号影响理解、竖排/多行按钮 | A |
| 交互流畅 | 精确点击语义 | zoom、mode、panel tests | 点击某项却循环到下一项，模式栏误触频繁 | B |
| 交互流畅 | 面板开合和返回 | 单一路由、outside/back close tests | 多面板叠加、关闭按钮难找、遮挡快门 | B |
| 功能可用 | 可触达功能有真实 owner | mode/session/device/media 对应 contract 和测试 | 假入口、只改文案、无 unsupported/degraded | C |
| 功能可用 | 关键模式行为一致 | mode declaration、capture policy、degrade tests | 模式切换后旧状态残留或请求未更新 | C |

| IO 通畅 | 录像金路径 | record start/stop 状态、文件或 sidecar、UI 反馈 | 开始/停止无反馈，输出不可打开 | D |

## 5. 职责切分

### 5.1 多模态高阶主控

负责最难 10% 与多模态限定工作：

1. **2.0 准入判据建模与跨域归因**：统一 UI、交互、架构、稳定性口径。
2. **视觉一致性高风险项仲裁**：设计逻辑、截图/录屏、代码行为不一致时做最终判断。
3. **输入输出链路端到端证据拼接**：从 UI 意图到媒体产物，识别跨层断点。
4. **最终 GO/NO-GO 评审会底稿**：可用于决策。

当前可先完成的高难 10% 是“判据、归因模型、证据矩阵、主控仲裁口径”。真实截图、录屏、APK 输出文件的多模态复核，必须等对应素材存在后执行，不能由文本代理假装完成。

### 5.2 非多模态并行 agent

按领域拆分执行：

- A 组：UI 结构一致性静态审查。
- B 组：交互流时序与可用性测试。
- C 组：功能可触达与有效性回归。
- D 组：IO 链路与媒体结果核验。
- E 组：稳定性、恢复与观测基线复验。
- F 组：发布准入清单与风险台账汇总。

## 6. 执行节奏

建议 3 天并行压缩：

- Day 1：并行采证 + 初版问题清单。
- Day 2：复测与归因 + 风险收敛。
- Day 3：准入评审 + GO/CONDITIONAL/NO GO 结论。

并行互斥规则：

- 同一时间只允许一个实现/审查 agent 作为 `MainActivity.kt` 和 `activity_main.xml` 的集成 owner。
- A/B/F 以文档和测试审查为主，不主动改生产代码。
- C/D/E 若发现 P0/P1，可以输出修复建议，但本批任务默认先交付审查报告，不直接修复。
- 任何 agent 都必须使用 `rtk` 运行命令。

## 8. 关键难点分析

1. **“看起来可用”与“可稳定交付”之间的证据鸿沟**：UI 截图美观不等于 media output 可信，单测通过也不等于真机路径通畅。
2. **UI 文档、实现、测试三者版本不同步导致假阳性**：以用户可见行为为第一真值，文档和测试都只是证据。
3. **多模式与多状态叠加下交互语法冲突**：横竖屏、面板、录制、保存、恢复同时出现时最容易暴露隐式状态 owner。
4. **真实设备相关问题本地难复现**：provider death、长稳恢复、thermal、相机服务重启必须标注外部依赖。
5. **性能/时延阈值缺少设备分层基线**：本地只做默认预算和诊断入口，不把设备矩阵空缺伪装成通过。

## 9. 交付物定义

- 一份 `2.0 Readiness 总报告`：汇总各域结果与判定。
- 一份 `P0/P1 风险清单`：附复现步骤、归属模块、证据、复测命令。
- 一份 `2 周收敛路线图`：若为 `CONDITIONAL GO` 或 `NO GO`。

## 10. 本轮主控初判

在未拿到最新 APK 截图、录屏、保存媒体文件、真机日志前，不能宣称 `GO`。基于当前文档化事实，合理默认是：

- 倾向启动 `CONDITIONAL GO` 审查流程，而不是直接 `GO`。
- 若 D 组证明拍照、录像、缩略图、保存链路存在 P0，则转 `NO GO`。
- 若 A/B 仅发现布局、文案、面板一致性 P2/P3，且 C/D/E 金路径通过，可维持 `CONDITIONAL GO` 或进入 `GO` 评审。
