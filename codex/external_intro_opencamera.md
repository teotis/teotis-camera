# OpenCamera 对外介绍文档

## 讲解框架

1. 先说明项目定位与重构动机，回答“为什么不是继续修旧工程，而是重新做一套新架构”。
2. 再说明 OpenCamera 的架构主线，重点讲清楚 Session Kernel、Mode Plugin、Device Adapter、Media Pipeline 如何分工协作。
3. 最后说明当前阶段成果、现阶段边界，以及后续会继续演进到什么方向。

## 项目概述

`OpenCamera` 是一个面向通用 Android 设备重新设计的相机应用工程。它不是对既有相机 App 的局部清理，也不是把旧工程换一套包名重新组织，而是围绕“复杂相机系统应该如何长期演进”这一问题，从架构边界开始重建。

这个项目的目标，不只是做出一个能拍照、能录像的 Demo，而是建立一套可以持续承载复杂模式、设备差异、异常恢复和后续功能扩展的相机工程底座。对于有 Android / Camera 背景的读者，可以把它理解为：一套以 `Session Kernel` 为中枢、由 `Mode Plugin`、`Device Adapter`、`Media Pipeline` 协同支撑的长期演进型相机架构实践。更准确地说，它的主线是“四层主链路”，而稳定性、可观测性与恢复能力则作为横切治理能力挂在整条链路之上，而不是再并列拆成新的业务层。

## 项目背景与重构动机

传统大型相机 App 往往会随着功能和机型适配不断累积复杂度。问题不在于“功能太多”，而在于功能、设备控制、状态管理、异常恢复和 UI 编排逐渐缠在一起，最终导致三个典型后果：

- 新模式和新特性越来越难接入，改一处容易影响多处。
- 设备差异和异常场景难以统一治理，问题出现后定位成本高。
- 工程对特定设备、特定历史实现、特定团队经验形成强依赖，长期维护成本持续上升。

`OpenCamera` 的重构动机，正是要从根上解决这些问题。它希望把“相机 App 是一个复杂实时系统”这件事认真落到工程边界上，而不是继续沿用普通业务 App 的组织方式。项目因此选择重新定义核心结构：把运行时主导权收回到统一会话内核，把模式能力改为插件式接入，把设备实现与媒体产出分别收敛到明确边界中。

## 设计目标

`OpenCamera` 当前明确追求以下几类目标：

- 建立稳定的会话主链路，让预览、拍照、录像、切模式、切镜头、权限变化与恢复流程由统一内核驱动。
- 让模式以插件方式接入，复杂模式可以扩展，但不会反向侵入主流程。
- 让设备差异有清晰的适配层，而不是散落在 UI、模式或各类业务代码里。
- 建立统一的拍摄与媒体管线，把拍摄请求、保存、缩略图、后处理和产物诊断串成同一条链路。
- 内建状态可观测与恢复能力，使问题能够被记录、定位和回归验证。
- 让工程具备长期演进基础，而不是随着功能增长重新滑回“大型耦合工程”。

## 核心架构思路

从对外介绍和面试表达的角度，OpenCamera 更适合概括为“四层主链路 + 一套横切治理能力”，而不是把所有能力拆成多个并列层级。四层主链路负责回答“业务意图如何进入系统、如何变成状态、如何落到底层设备、如何变成最终媒体产物”；横切治理能力则负责回答“出了问题如何观察、恢复和回归验证”。

### 1. 以 Session Kernel 作为运行时中心

OpenCamera 的核心不是单个模式，也不是 UI 层，而是一次 `CameraSession`。项目把“当前相机运行状态”的解释权集中到会话内核中，由它统一管理生命周期、权限状态、预览状态、当前模式、当前拍摄任务以及恢复相关决策。

这意味着：

- UI 主要负责渲染状态和分发用户意图。
- 预览绑定、拍摄执行、停止录制、异常恢复等关键动作由 session 统一裁决。
- 外部事件先转换成 session 可消费的信号，再进入主链路处理。

这条主线的价值在于，把相机系统中最容易失控的“谁在真正驱动运行时”这个问题收口到一个中心点，避免 App Shell、协调层、模式层各自维护半套运行时逻辑。

### 2. 用 Mode Plugin 承载模式差异，而不是复制整套相机流程

OpenCamera 将 `PHOTO`、`VIDEO`、`NIGHT`、`PORTRAIT`、`DOCUMENT`、`PRO` 等模式组织为插件。模式的职责不是直接操作底层设备，而是声明自身的：

- UI 规格与交互动作
- 拍摄策略
- 模式特有状态
- 产出偏好和能力降级方式

这样做的重点不是“把模式拆成多个类”这么简单，而是把模式从底层执行细节中解耦出来。模式只表达“我要什么样的拍摄行为”，真正的执行、状态推进和后处理编排由下层统一完成。

### 3. 用 Device Adapter 隔离 CameraX / 设备差异

当前工程通过设备适配层把抽象拍摄请求翻译到实际设备执行上，并以 `DeviceCapabilities`、`DeviceGraphSpec`、拍摄请求翻译等方式描述设备能力与执行条件。现阶段的验证实现以 CameraX 为主，但架构上已经明确把 CameraX 视为适配实现，而不是未来唯一的系统边界。

这让项目在两个方向上受益：

- 模式和会话层不需要直接理解底层 API 细节。
- 后续如果引入更深的设备能力、厂商能力或更复杂的执行策略，扩展点是清楚的。

### 4. 用统一 Media Pipeline 管理产物生命周期

OpenCamera 不是让每个模式自己拼拍照保存逻辑，而是把拍摄请求先收敛为统一 `Shot`，再进入统一媒体管线，覆盖：

- 拍摄请求生成
- 媒体保存任务
- 缩略图来源
- 输出句柄
- 后处理入口
- 产物诊断信息

这条链路已经能承载静态图片和视频两类输出，并允许后处理以流水线方式叠加到最终媒体结果上。它的意义在于，把“模式策略”和“媒体产物处理”拆开，减少功能扩展时的重复实现。

### 横切治理能力：可观测与恢复内建，而不是事后补日志

相机工程最怕“出了问题只能猜”。OpenCamera 现阶段已经把状态轨迹、关键动作和预览指标纳入统一会话链路，并将预览恢复、首帧、权限阻断等关键信号纳入状态模型。它不是完整稳定性体系的终点，但已经不是传统工程里那种只能靠零散日志拼时序的状态。

## 关键模块职责划分

如果只看主业务主链，OpenCamera 的架构摘要可以统一为：

- 模式层：负责产品语义与拍摄策略表达。
- 会话内核层：负责全局状态、规则和运行时决策。
- 设备适配层：负责能力协商与底层设备执行。
- 媒体管线层：负责拍摄结果到最终媒体产物的统一组织。

在这四层之外：

- App Shell / UI 更适合作为宿主与渲染层理解，而不是并列主链路层。
- 稳定性、可观测性、恢复和自动化验证更适合作为横切治理能力理解，而不是独立业务层。

### Session Kernel

负责统一接收意图、维护会话状态、推进主状态机，并发出预览绑定、执行拍摄、停止拍摄等效果。它是四层主链路中的运行时中枢，也是当前行为决策的唯一中心。

### Mode Plugin

负责定义模式差异，包括模式 UI、模式状态、拍摄策略、模式特有的参数选择与能力降级。它不直接控制设备，也不单独接管恢复逻辑，更像是四层主链路中的产品语义表达层。

### Device Adapter

负责把统一拍摄请求翻译为底层设备执行行为，并向上层回传预览、拍摄开始、拍摄完成、失败等设备事件。当前主要承担 CameraX 适配与设备能力探测，是四层主链路中的平台能力协商与执行落地层。

### Media Pipeline

负责把拍摄结果组织成稳定的媒体产出链路，包括保存、缩略图、媒体输出句柄、后处理和诊断信息补充。它是“从一次拍摄请求到最终产物”之间的统一执行通道，因此后处理也被视为媒体管线内部能力，而不是再拆成并列大层。

### App Shell / UI

负责依赖装配、宿主生命周期接入、状态渲染和用户输入分发。它不是隐藏的第二套状态机，也不旁路驱动相机主流程；从架构摘要上更适合作为宿主层，而不是与四层主链路并列。

## 与传统相机工程写法相比的主要改进

### 从“多处共同驱动运行时”转向“单一会话内核驱动”

传统相机工程常见的问题，是 Activity、Coordinator、模式控制器、设备层都在解释当前状态，逐步形成多个影子状态机。OpenCamera 的改进点，是把运行时主导权收回到 session，让其他层更多成为边界适配者。

### 从“模式直接碰设备细节”转向“模式声明策略，底层统一执行”

在很多历史工程里，模式越复杂，越容易把底层设备逻辑、保存逻辑和 UI 逻辑揉到一起。OpenCamera 则把模式约束在策略表达层，使复杂模式可以扩展，但不必重新复制拍摄主链路。

### 从“拍照与保存分散实现”转向“统一 Shot / Media Pipeline”

传统写法常见每个模式各自维护一套保存、缩略图、后处理逻辑，时间久了差异越来越大。OpenCamera 把这些流程收敛到统一管线中，提高了复用性，也降低了后续扩展水印、画幅、风格渲染等能力时的接入成本。

### 从“功能优先”转向“横切治理能力内建”

项目不是只看界面上是否出现一个按钮，而是把状态边界、恢复路径、失败诊断和测试可回归作为横切治理能力内建到主链路里。对相机这种高状态复杂度系统，这类工程价值往往比单个功能点更关键。

### 从“绑定特定旧工程经验”转向“面向通用 Android 设备的开放底座”

OpenCamera 的一个重要方向，是从历史上深绑定单一设备体系的工程思路中抽离出来，建立可面向更广泛 Android 设备持续演进的能力边界。

## 当前阶段成果

### 已完成

- 已建立以 `CameraSession` 为中心的会话主链路，预览、拍照、录像、切模式、切镜头、权限变化与预览宿主变化都已进入统一 session 驱动流程。
- 已完成模式插件化基础，当前具备 `PHOTO`、`VIDEO`、`NIGHT`、`PORTRAIT`、`DOCUMENT`、`PRO` 六类代表性模式。
- 已形成设备能力与设备图的基础抽象，并通过 CameraX 适配层打通真实预览、静态拍摄和视频录制。
- 已形成统一拍摄与媒体管线，覆盖拍摄请求、保存任务、缩略图来源、媒体输出句柄和统一后处理入口。
- 已落地一批可见产出能力，包括静态图片水印、图片风格处理、文档自动裁边、画幅裁切、人像渲染，以及视频侧的字幕式水印产物补充。
- 已具备夜景模式的多帧执行闭环、视频 torch 控制、静态拍摄质量与分辨率选择等代表性能力，用于验证架构能够承载复杂模式策略。
- 已补入较完整的单元测试与构建级验证，覆盖 session、device、media pipeline、主要 post-processor 和部分 UI 渲染决策。

### 正在建设

- 稳定性治理正在从“结构正确”继续向“长期可回归验证”推进，重点会落在更系统化的恢复、异常场景和自动化验证能力上。
- 设备适配边界已经建立，但更深层的设备差异治理与更强的设备能力利用还会继续演进。

### 当前项目成熟度判断

从工程阶段看，OpenCamera 已经越过“架构概念验证”阶段，进入“核心主链路成立、代表性模式可运行、能力边界基本成型”的阶段。它已经可以清晰展示新架构的价值，但仍处于持续打磨和扩展期，而不是最终产品形态。

## 当前边界与后续演进方向

### 当前边界

- 当前工程已经具备完整主链路与代表性模式，但整体重点仍然是“架构底座成立”和“能力边界验证”，不是覆盖所有成熟商用相机特性的最终版本。
- 现阶段很多关键结论已经通过本地测试、构建和代码结构得到验证，但面向复杂真机环境的稳定性验证体系仍会继续加强。
- 部分复杂能力已经打通执行闭环，但在最终成像质量、设备深度适配和更细粒度体验治理上，仍有进一步提升空间。

### 后续演进方向

- 继续补强稳定性治理，包括首帧、切模式、切镜头、权限中断、恢复重建、保存失败等关键路径的自动化回归能力。
- 继续深化设备适配层，使模式能力与底层实现进一步解耦，为更复杂设备能力和差异化接入留出空间。
- 继续完善统一媒体管线，把更多真实特性接入标准化后处理与产物治理链路。
- 继续扩展复杂 feature，但坚持通过模式插件、会话服务、设备适配层和媒体管线接入，避免回到全局耦合式增长。
- 在代表性模式稳定后，进一步把架构验证从“能承载复杂模式”推进到“能长期支撑复杂产品演进”。

## 一段适合口头讲解的简版总结

`OpenCamera` 可以理解为一套重新设计的 Android 相机工程底座。它不是简单重写一个相机 App，而是针对传统大型相机工程里常见的状态混乱、模式耦合、设备差异难治理和恢复链路脆弱等问题，重新建立了一条“四层主链路 + 横切治理能力”的架构主线：模式层负责产品语义，会话内核负责运行时决策，设备适配层负责能力协商与执行落地，媒体管线负责统一产物组织；稳定性、可观测性与恢复能力则贯穿整条链路。当前项目已经完成会话内核、模式插件体系、设备适配层和统一媒体管线的基本落地，并通过拍照、录像、夜景、人像、文档、专业模式等代表性能力验证了这套架构确实能够承载复杂相机场景。下一步工作的重点，不是推倒重来，而是在这条主线上继续补强稳定性、设备适配深度和更完整的功能演进能力。
