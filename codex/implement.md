### 执行目标

在**当前任务范围**内持续推进重构。  
先判断当前所处阶段，再围绕当前阶段内**最高价值、最可验证、最影响阶段完成度**的问题，持续执行：

**识别 → 实施 → 验证 → 修正 → 更新文档 → 再识别**

**一次收到“推进任务”命令后，不因只完成一个 patch、一个小点、一次测试通过、一次文档更新而自动停止。**  
只有命中停止条件时，才允许停止并汇报。

禁止无边界扩散、低价值整理、顺手大改、假推进、跳过验证叠加改动。

---

### 执行任务时，如何确定当前总体目标是什么

用户明确指令最优先。  
若用户没有指定本轮只做某个点，则默认执行如下判断逻辑：

1. 阅读 `codex/plan.md`
2. 阅读 `codex/prompt.md`
3. 阅读 `codex/documentation.md`
4. 结合代码现状，判断：
   - 当前在哪个阶段 / milestone
   - 当前阶段的核心目标是什么
   - 当前阶段距离完成还缺什么
   - 当前阶段当前最大的阻塞点 / 最大缺口是什么
   - 当前最值得优先完成的闭环是什么

若文档与代码现状不一致，以**代码真实状态 + 当前验证结果**为准，并同步修正文档判断。

---

### 当前阶段完成度的判断标准

`plan.md` 中对应阶段达到 **95% 以上**，指的是：

- 当前阶段核心目标已经成立
- 当前阶段关键产物已经基本落地
- 主路径已经打通，关键验证通过
- 主要 P0 / P1 级问题已收敛
- 剩余问题主要为低价值、低收益、非关键路径项
- 当前代码结构、接口边界、验证现状，已足以支撑下一阶段继续开发

**95% 以上不等于自动进入下一阶段。**

只有在以下条件同时满足时，才视为当前阶段已经具备“可进入下一阶段”的资格：

1. 当前阶段已达到 95% 以上
2. 已完成当前阶段的工程复盘与加固
3. 当前阶段遗留问题已收敛到可接受范围
4. 用户明确允许进入下一阶段

**每次跨阶段，都必须重新获得用户确认。用户一次确认，只对一个阶段生效。**

---

### 单次任务执行原则

- 不得顺手大改命名和格式化全仓
- 不得为了“显得在持续运行”而做低价值改动
- 不得扩大到未指定模块
- 不得跳过验证直接叠加改动
- 不得把一次任务拆成大量低价值碎步来制造进度
- 不得在没有形成有效闭环前，把任务宣称为完成
- 不得把“写了文档 / 写了 TODO / 做了计划”当成真实推进
- 不得因为完成一个 patch 就自动停止

在**不跨阶段、不扩范围**的前提下，允许连续处理多个**强相关**子任务，直到完成一个又一个可验证闭环，或命中停止条件。

执行单位不是“孤立的一处小改动”，而是：

**当前阶段内最高价值、可实施、可验证的一个问题簇 / 一个最小可闭环子目标。**

为完成同一闭环，允许一并进行以下强相关工作：

- 局部实现改动
- 必要的接口适配
- 必要的测试补齐
- 必要的验证脚本或校验修正
- 修复本轮改动直接暴露的问题
- 为本轮闭环成立所必须的少量结构整理

但不得扩散到与当前闭环无关的问题。

---

### 默认执行姿态

收到“继续推进 / 持续推进 / 根据 codex 文件继续做”之类命令后，默认行为不是：

- 只做一个 patch 就停
- 只做一个点就回来汇报
- 只更新一次文档就结束

默认行为应当是：

**在当前阶段内，连续完成多个高价值闭环，直到：**
- 当前阶段已经满足停止条件
- 或出现必须由用户决策 / 协助 / 授权的问题
- 或继续推进已不再有明显收益

也就是说：

**“持续推进”默认意味着连续循环，不意味着只推进一步。**

---

### 每轮固定流程

#### 1. 识别状态

先明确：

- 当前在哪个阶段 / milestone
- 当前阶段完成度大致处于什么区间
- 当前阶段是否已完成工程复盘与加固
- 当前最大缺口是什么
- 当前最值得优先完成的闭环是什么
- 为完成这个闭环，最少需要涉及哪些改动
- 本轮验证应该跑哪些最小验证、哪些必要验证

若当前问题很多，优先处理：

1. 阻塞主路径的问题
2. 影响阶段完成度的问题
3. 影响后续架构继续演进的问题
4. 当前验证无法通过的问题
5. 本轮刚引入的问题

---

#### 2. 实施闭环内改动

- 只处理当前最有价值的一个闭环
- 改动尽量局部、克制、可回溯
- 可以包含多个强相关改动，但都必须服务于同一个闭环
- 不处理与本轮目标无关的问题
- 不为了“顺手”扩展到旁支模块

若当前闭环未完成，不得中途转去做另一个低优先级点。

---

#### 3. 执行验证

每完成一个闭环，都必须验证。  
验证结果优先于主观判断。

验证顺序：

1. 最小验证
2. 必要的编译 / 测试 / 运行级验证
3. 检查关键日志、关键状态、关键产物
4. 检查是否引入新的明显回归
5. 判断该闭环是否真实成立

不能验证的改动，不算完成。

---

#### 4. 失败处理

若验证失败：

- 先判断是否为本轮引入的问题
- 若是，先修复本轮问题
- 修复后重新执行最小验证和必要验证
- 不得带着失败继续叠加新改动
- 若失败根因不明，不得假设通过
- 应继续定位，直到问题收敛、结论明确，或命中停止条件

---

#### 5. 闭环后继续判断

每完成一个闭环后，**不应默认停止**。  
必须继续判断：

- 当前阶段内是否仍存在明确高价值缺口
- 是否还能在当前范围内继续形成下一个可验证闭环
- 是否还有阻塞当前阶段完成的 P0 / P1 问题
- 是否已经命中停止条件

若仍存在明确高价值缺口，且未命中停止条件，则直接进入下一轮循环。  
若没有，则停止。

---

#### 6. 更新 `documentation.md`

`documentation.md` 不要求每个微小 patch 都更新一次。  
但以下情况必须更新：

- 完成一个有实质价值、且已经验证通过的闭环后
- 当前阶段进度百分比发生实质变化后
- 当前阶段的关键判断发生变化后
- 本次任务停止前

上一轮旧记录不应直接删除。  
应保留对后续任务仍有价值的信息，并进行合并、去重、无损简化。  
对已经失效、重复、低价值、仅服务于历史中间态的问题，可明显压缩。

最新更新记录必须包括：

- 当前阶段 / milestone
- 当前阶段进度百分比
- 当前阶段是否已完成工程复盘与加固
- 本轮 / 本次任务完成的闭环内容
- 本轮关键技术决策
- 本轮验证结果
- 当前遗留风险
- 下一步建议

除对工程本身的建议外，若对 `plan.md`、`prompt.md`、`implement.md` 有优化建议，应单独归档在新标题下，提醒用户决定是否采纳。  
**不得擅自修改这些指导文件的核心规则。**

`documentation.md` 必须长期维持以下骨架：

当前状态

- 里程碑
- 阶段进度
- 当前阶段判断
- 当前验证基线
- 当前遗留风险
- 下一步建议

最近有效闭环

日期：闭环名

- 目标
- 核心结果
- 验证
- 结论

历史归档（精简版）

阶段摘要

- 结论
- 关键事实
- 留存影响

### 阶段完成后的额外要求

当前阶段达到 95% 以上后，**不等于自动进入下一阶段**。  
必须先确认是否已完成**工程复盘与加固**。

复盘与加固至少要确认：

- 当前阶段目标是否真正闭环
- 关键路径是否稳定
- 验证是否覆盖主要路径和关键边界
- 是否存在阻碍下一阶段的遗留问题
- 当前代码结构是否足以支撑下一阶段继续开发
- 当前阶段技术债是否已经收敛到可接受范围
- 当前阶段已有实现是否存在“表面能跑、实则脆弱”的问题

若以上未满足，则不得进入下一阶段。

---

### 何时重新分析“当前总体目标”

出现如下情况时，应重新分析当前总体目标，再决定是否继续当前阶段内循环：

1. 当前范围内已无明确高价值改动
2. 继续修改只会带来很小边际收益
3. 连续两轮未形成新的有效闭环
4. 当前问题表象较多，但缺少清晰主线
5. 当前验证结果显示，原优先级判断可能有误
6. 当前文档描述与代码现实偏差明显

重新分析总体目标，不等于自动跨阶段。  
若仍处于当前阶段，则应重新识别当前阶段内最高价值闭环，继续推进。

---

### 任务停止条件

出现以下情况之一，任务停止：

1. 当前阶段已经达到可进入下一阶段的条件  
   （即：当前阶段达到 95% 以上，且复盘加固完成），但尚未获得用户允许跨阶段的明确指令  
   **用户一次确认，仅对一个阶段生效；每次跨阶段都必须再次确认**
2. 出现重大关键设计分歧或重大不确定性，必须由用户决策
3. 出现必须依赖用户协助、权限、环境、外部资源或信息补充，才能继续推进的情况
4. 当前阶段范围内，已无明确高价值、可实施、可验证的剩余工作
5. 继续修改将明显引入大于收益的复杂度、风险或技术债
6. 连续推进已无法形成新的有效闭环，且剩余工作主要为低价值项

---

### 执行时的硬性要求

- 始终以“完成当前阶段”为边界，不跨阶段抢跑
- 始终以“完成一个可验证闭环”为最小推进单位，而不是堆零散 patch
- 始终以验证结果而不是主观感觉判断进度
- 始终优先处理阻塞关键路径、影响阶段完成度、影响后续开发的问题
- 始终避免审美型重构、展示型改动、低收益清理
- 始终在每个有效闭环后继续判断是否还应推进下一闭环
- 未命中停止条件前，不得因为只完成一个小改动、一个 patch、一次局部验证、一次文档更新而自动停止
