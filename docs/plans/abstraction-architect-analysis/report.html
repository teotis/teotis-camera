执行摘要

OpenCamera 项目整体架构设计良好，四层架构（Mode Plugin → Session Kernel → Device Adapter → Media Pipeline）的边界在大多数地方得到了正确遵守。核心发现：

  • 1 个结构性问题：core:media ↔ core:capability 循环依赖（源码层面未使用，可直接删除）
  • 4 个改进机会：构建配置残留、依赖扇出过宽、具体类耦合、macOS 资源叉文件
  • 9 个合规确认：Session 状态所有权、feature 模块隔离、core 内部依赖方向等均正确

扫描范围：app/ (55 .kt files), core/ (109 files across 7 modules), feature/ (14 files across 8 plugins)

CRITICAL 关键问题

C-1 core/media ↔ core/capability 循环依赖
违规类型模块级循环依赖
涉及模块core:media ↔ core:capability
违反规则核心模块之间不应存在循环依赖，破坏可测试性和编译隔离
证据：
  • core/media/build.gradle.kts 声明 implementation(project(":core:capability"))
  • core/capability/build.gradle.kts 声明 implementation(project(":core:media"))
  • 源码层面：core/capability 的 CapabilityGraphResolver.kt 确实导入 core.media.MediaProcessorAvailability
  • 源码层面：core/media 源码中零导入 core.capability 包的任何类
影响：编译时耦合循环，无法单独测试任一模块。media → capability 的依赖在源码层面完全未使用。
建议修复：从 core/media/build.gradle.kts 移除 implementation(project(":core:capability"))。这是一行改动，零风险。

WARNING 改进机会

W-1 CameraXCaptureAdapter 导入 7 个 core 实现类
文件app/.../camera/CameraXCaptureAdapter.kt
违规类型具体实现类耦合
涉及的具体类：
  • :61 — DefaultDeviceShotRequestTranslator
  • :76 — MultiFrameCaptureExecutionPlanner
  • :78 — MultiFrameTemporaryOutputTracker
  • :96 — CompositeMediaPostProcessor
  • :108 — MultiFrameMergePlaceholderPostProcessor
  • :109 — PipelineMetadataPostProcessor
  • :111 — ShotExecutor
影响：adapter 与 core.device 和 core.media 的具体实现强耦合。不过 adapter 作为硬件桥接层，直接使用实现类在实践中可接受（当前不存在对应接口）。其中 ShotExecutor 和 CompositeMediaPostProcessor 作为核心媒体处理管道，值得考虑通过接口注入。
W-2 core/media 声明了两个未使用的模块依赖
文件core/media/build.gradle.kts
违规类型构建配置残留
证据：
  • api(project(":core:device")) — media 源码中零导入 core.device 包
  • implementation(project(":core:capability")) — media 源码中零导入 core.capability 包
影响：传递依赖膨胀（api scope 会暴露给所有 media 消费者），增加编译时间。
建议修复：将 api(project(":core:device")) 改为 implementation 或直接删除；删除 implementation(project(":core:capability"))（与 C-1 同一修复）。
W-3 core/mode 的 api 依赖扇出过宽
文件core/mode/build.gradle.kts
违规类型过度传播
证据：mode 以 api scope 依赖 5 个 core 模块（device, media, settings, effect, capability），但 7 个源文件只导入 20 个唯一符号。使用 api 意味着所有 mode 消费者都传递性获得这 5 个模块的全部公开 API。
影响：session 模块已经是 mode 的消费者且直接依赖这 5 个模块，实际影响有限。但如果未来有新消费者只依赖 mode，会不必要地获得所有传递 API。
建议修复：将不需要传递的依赖从 api 改为 implementation。
W-4 core/session 包含 macOS 资源叉文件
文件core/session/.../._CaptureRecordingSessionProcessor.kt
违规类型源码目录中的二进制/元数据文件
影响：macOS ._ 前缀文件是 extended attribute 的资源叉，不应纳入源码控制。可能在某些构建工具中造成解析异常。
建议修复：删除文件并添加 .gitignore 规则 ._*。

PASS 架构合规确认

I-1 AppContainer.kt 作为 DI 组装点（预期行为）

AppContainer 是 DI/组装根，导入所有具体实现类是正确的。它导入了 DefaultCameraSession、InMemorySessionTrace、CompositeMediaPostProcessor、ShotExecutor、ModeRegistry、CapabilityGraphResolver、EffectCapabilityResolver、PreviewEffectAdapter 和 8 个 feature mode plugins。这是唯一应该知道所有具体实现的文件。架构正确。

I-3 CameraX/Camera2 导入集中且可控

Camera2 导入仅在 CameraXCaptureAdapter.kt 一个文件（7 个 import）。CameraX 导入集中在 CameraXCaptureAdapter.kt（28 个 import），其余为 PreviewView（UI 组件）或 ImageProxy（帧处理）。feature/ 模块零 CameraX/Camera2 导入。

I-4 feature/ 模块完全遵守架构契约

所有 8 个 mode plugin 检查结果：CameraX/Camera2/HAL 导入零 ✅ · app/ 层导入零 ✅ · core.session 导入零 ✅ · 所有导入来自 core.mode, core.device, core.media, core.effect, core.settings（均为契约/值类型）。"Mode plugins must not call CameraX/Camera2/HAL directly" 规则完全遵守。

I-5 Session 状态所有权边界完全清洁

逐文件验证：DefaultCameraSession 是唯一拥有者 ✅ · CameraSessionCoordinator 只 dispatch intent ✅ · SessionSettingsManager 只 dispatch intent ✅ · SessionUiRenderModel 纯只读 ✅ · CameraXCaptureAdapter 不读不写 state ✅。架构遵循 intent-driven 模式，无隐藏 session kernel。

I-6 core 内部依赖方向正确

core.mode → core.session: 无导入 ✅ · core.device → core.session: 无导入 ✅ · core.media → core.session: 无导入 ✅ · core.settings → 任何 core: 无导入 ✅ · core.session 的 55 个跨模块符号导入全部指向 mode/device/media 的契约类型（接口、sealed interface、data class、enum）。

I-7 core.session 的公开 API 设计正确

CameraSession 接口仅暴露：state: StateFlow<SessionState>（只读）、effects: Flow<SessionEffect>（只读）、suspend fun dispatch(intent: SessionIntent)（唯一变更入口）。_state: MutableStateFlow 为 private，intent 通过 channel 顺序处理，无并发状态变更。

I-8 75 个 app/ 源文件中仅 3 个存在具体类导入

AppContainer.kt (~20 个，预期) · CameraXCaptureAdapter.kt (7 个，适配器层可接受) · SessionPreviewRenderModel.kt (1 个，轻微)。其余 72 个文件仅导入接口、enum、data class、sealed interface 或 top-level function。
