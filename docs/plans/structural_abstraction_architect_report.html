Executive Summary

这五类问题都“有”，但成熟度不同。最有用户价值和研发价值的第一步是补 feature module 的直接单元测试，因为它能立刻把模式插件输出的 EffectSpec、PostProcessSpec、metadata tags 和 session events 固定下来，为后续拆大类和收敛 metadata 桥提供安全网。

最高杠杆的结构机会是把 Effect/Capture 中的渲染意图变成更完整的 typed payload，从而逐步删除 post-processor 里反复读取 customTags 的逻辑。Document-batch 则是一个真实但不应急着整体迁移的试验田：先抽出 session 内部 reducer/extension seam，再决定是否允许 mode-owned command handler。

大型文件拆解有效，但优先级应低于测试安全网和表示收敛。它适合按已存在的自然职责拆，不适合按行数机械切文件。

Structural Pressure Map

  • Representation duplication: EffectSpec, RenderRecipe, PostProcessSpec, MediaMetadata.customTags 同时表达一部分渲染意图。
  • Conversion glue: mode plugins 调 EffectBridge，然后自行补 custom tags；post-processors 再反向解释 tags。
  • Orchestration hotspot: CameraXCaptureAdapter 同时承担 CameraX runtime、capture、video、preview bitmap、live/photo feedback、postprocess guard 等职责。
  • Boundary friction: app 层有 Android-bound render IO 是合理的，但领域 render decisions 混在其中，增加迁移难度。
  • Mode-specific accumulation: Document-batch 当前为了避免 UI hidden state 放入 session，这是合理的局部选择，但也证明 session 需要可控 extension seam。

Feature Plugin Direct Tests

Classification: Validated structural opportunity Structural Leverage: High Transition Burden: Low Risk: Low Confidence: High

Current symptom: feature modules encode mode behavior but have no direct tests. Behavior is partly covered indirectly by app/session tests, which makes refactoring and metadata migration harder than necessary.

Hidden invariant: A mode plugin should be testable as a pure contract emitter: selected runtime settings in, capture strategy/effect spec/session events out.

Complexity deleted: Reduces reliance on app/session integration tests for mode plugin regressions and creates a safe base for later Effect/Capture migration.

Disproof signal: If plugins require Android runtime or real camera dependencies to instantiate, first extract test harness seams instead of writing brittle tests.

Typed Effect Payload Through Capture

Classification: Promising but needs narrow pilot Structural Leverage: High Transition Burden: Medium Risk: Medium Confidence: Medium High

Current symptom: EffectBridge writes both typed-derived tags and a reduced PostProcessSpec. Post-processors still parse customTags for frame ratio, watermark style, portrait/document markers, and perceptual recipes.

Hidden invariant: Preview, capture, saved render, thumbnail, and diagnostics should consume one render intent where possible; string metadata should be persistence/compatibility, not the main intra-process API.

Meaningful differences preserved: EXIF/persisted tags remain string metadata; Android-bound rendering remains in app; core/effect does not learn CameraX or Bitmap.

Pilot: Pick one small tag family, preferably frame ratio or selfie mirror, carry it as typed payload into ShotRequest/ShotResult, and delete exactly one customTags reparse path.

Disproof signal: If migration adds another representation without deleting reparse logic, stop and revert to documentation plus tests only.

Mode-Specific Session Extension Seam

Classification: Promising but unproven hypothesis Structural Leverage: Medium Transition Burden: Medium Risk: Medium Confidence: Medium

Current symptom: Document-batch state and intents are in core session, and DefaultCameraSession handles them directly.

Why this is not simply wrong: The original choice avoided hidden UI ownership and made saved-media state testable. That is valuable. The risk is future repetition for every mode-specific workflow.

Pilot: Extract document-batch operations into an internal reducer or session extension object first. Keep public session state behavior unchanged. Only after tests pass should the team evaluate a mode-owned command extension.

Disproof signal: If only document mode needs this for the foreseeable future, a local reducer is enough; a generic extension framework would be premature.

Responsibility-Based Large File Decomposition

Classification: Pragmatic sequencing problem Structural Leverage: Medium Transition Burden: Medium High Risk: Medium Confidence: High that problem exists

Current symptom: Four central files are large enough to impair review and increase merge conflict risk.

Recommended sequence: Do not split by line ranges. Split along already visible responsibilities: CameraX provider/preview lifecycle, still capture, video recording, metering/brightness, live/thumbnail helpers; render-model composers by cockpit/preview/settings/diagnostics/document batch; session reducers by preview/recovery, capture/recording, mode/device control.

Proof of value: Each split should move tests with the extracted unit or add focused tests. A split that only creates wrappers without reducing caller knowledge should be rejected.

Rejected Or Deferred Abstractions

  • Generic event bus: no deletion proof; risks hiding session ownership.
  • :core:shared bucket module: likely worsens boundaries unless a concrete shared invariant is identified.
  • State machine framework: session already has tested transitions; framework migration burden is not justified by the inspected evidence.
  • Move all post-processing to core: false abstraction risk because bitmap, EXIF, URI, and file IO are Android-bound.
  • Parallel post-processing for performance: needs measured bottleneck data first.

Transition Handoff Brief

Preferred first breakthrough: Add direct feature-mode plugin unit tests. This is low-risk, directly valuable, and creates the validation floor for all other work.

Second step: Design a narrow typed render intent pilot that removes one customTags reparse path. Do not introduce a generic abstraction unless it deletes existing branches.

Third step: Extract document-batch operations into an internal reducer as a reversible step. Keep public behavior and existing tests stable.

Authorization boundary: This report is analysis only. Production code changes require a separate approved implementation plan.
