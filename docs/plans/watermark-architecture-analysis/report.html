Watermark Architecture Analysis

2026-06-03 review of the OpenCamera watermark system after real-device feedback: blur-four-border does not look convincingly blurred, while the frame/bottom-bar style preview is relatively acceptable.

Immediate Bug

drawContentAwareEdgeBorder uses downsample-upscale strips. This can soften large regions but is not a true blur kernel, so high-detail edges can still read as stretched content rather than blurred content.

Testing Gap

Current tests mainly prove that border pixels are derived from source colors and that simple gradients are smooth. They do not prove high-frequency detail is suppressed.

Structural Gap

Watermark templates are not canonical render contracts. Settings, preview hint, metadata bridge, output renderer, and tests each reinterpret the template id.

User Intent Recovered

Historical sessions show Watermark 2.0 was approved around four concrete directions: professional parameter bottom bar, minimalist text, blurred four-border, and reversible watermark. The relevant product pattern was not "copy a vendor", but use vivo/OPPO/Apple as references for clear selected state, visible preview affordance, professional shooting metadata, and non-intrusive image identity.

Why Bottom Bar Feels Better

The bottom-bar preview is mostly a layout projection: a bar, labels, and background. It does not need real source-image sampling to be understandable.

Why Four-Border Feels Wrong

The four-border template promises a content-aware optical effect. The preview cannot sample content, and the saved renderer does not use a real blur kernel, so the user's expected invariant is broken twice.

Abstraction Diagnosis

The inherited vocabulary says "template id", but the domain object users perceive is a watermark render contract: visual family, allowed style controls, preview projection, output projection, fidelity level, and acceptance tests. Today only the id is shared, while the rest is reconstructed in many places.

Recommended Transition

  1. Add a high-frequency pixel regression test for blur-four-border. It should fail when borders preserve edge detail too strongly.
  2. Replace downsample-upscale with a deterministic separable blur helper for edge strips, scoped only to the saved JPEG renderer.
  3. Strengthen preview hint honestly: keep it lightweight, but make it visibly express "blurred border area" rather than just a pale frame.
  4. Introduce a small WatermarkTemplateRenderContract helper after the local fix, not before. It should derive allowed controls, default style, preview shape, and renderer selection from one place.

Disproof Signals

This analysis is wrong if a high-frequency saved-render test already shows strong suppression of edge detail, or if real-device screenshots show the saved JPEG is actually blurred and the complaint is only about live preview. Current evidence points to both saved-render quality and preview affordance being too weak.
