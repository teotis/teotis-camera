OpenCamera 性能监控全量分析报告

分析日期: 2026-05-30 | 分析范围: 全量源码 (app/ + core/ + feature/) | 分析重点: 性能监控架构适配性

核心结论

当前项目已具备中等水平的性能监控基础设施，覆盖了预览首帧延迟、拍摄管道耗时、热状态和资源预算等关键维度。但存在5 个结构性缺口和3 个架构层面的适配问题，导致无法形成端到端的性能闭环。建议按优先级分 3 个阶段补齐。

2.2 详细缺口说明

G1 快门按下→拍照执行延迟

现状: DefaultCameraSession 在 ShutterPressed 时记录 capture.shutter.pressed 并设置 shutterPressedAtElapsedMillis，但该值仅用于 UI 展示，未作为独立性能 metric 被追踪。

应当补充: 记录从 shutterPressedAt 到 executeShot 入口 requestedAt 的差值，作为 "shutter-to-device" latency 单独 trace。

影响: 这是用户感知最直接的性能指标——按快门到看到拍照反馈的等待时间。

G2 模式切换耗时

现状: 仅记录 mode.switched 事件（切换完成后），无开始时间点。

应当补充:

  • mode.switch.started — 记录 SwitchMode intent 处理开始时间
  • mode.switch.completed — 已有，补充耗时差值
  • 子步骤: mode.switch.unbind → mode.switch.bind → mode.switch.first.frame

影响: 模式切换是高频操作（Photo↔Video↔Portrait 等），用户对卡顿敏感。

G3 镜头切换耗时

现状: 仅记录 lens.switched 事件，无耗时。

应当补充: 类似 G2，记录前后摄切换的完整耗时链。

G4 各后处理器单独耗时

现状: CompositeMediaPostProcessor 链式调用 7 个处理器，但只有整体 postProcessCompletedAt 时间戳，无法区分哪个处理器是瓶颈。

后处理器列表:

  • MultiFrameMergePlaceholderPostProcessor
  • DocumentAutoCropPostProcessor
  • PhotoFrameRatioPostProcessor
  • PortraitRenderPostProcessor (ML Kit segmentation)
  • PhotoAlgorithmPostProcessor
  • PhotoWatermarkPostProcessor
  • PhotoSelfieMirrorPostProcessor
  • PipelineMetadataPostProcessor

应当补充: 每个处理器的 start/end 时间戳，至少以 pipelineNotes 形式记录。

G5 文件 I/O 耗时

现状: 完全无文件写入耗时监控。

应当补充: 在 MediaSaveTransaction 中记录文件写入开始和完成时间。

G6 预览 FPS / 丢帧率

现状: PreviewSceneBrightnessMonitor 每 800ms 采样亮度，但不监控帧率。MlKit 场景掩码源有 framesDropped 计数但仅用于 Log.d。

应当补充:

  • 预览帧率采样（可复用 brightness monitor 的采样周期）
  • CameraX PreviewView 的 FrameDropListener（如可用）

三、架构适配性评估

3.1 适配良好的部分

✓ 分层架构清晰

性能监控代码分布在正确的架构层：core:session 负责 trace 和 diagnostics，app:camera 负责设备级计时，core:media 负责资源预算。没有跨层耦合。

✓ 预算系统设计合理

ThermalBudgetBridge + ResourceAdmissionPolicy 形成了完整的资源准入控制链，能根据 thermal state 和 performance class 动态降级各特性。这是优秀的防御性设计。

✓ RuntimeIssueMonitor 组合模式

CompositeRuntimeIssueMonitor 允许灵活组合多个监控器，扩展性好。

3.2 需要优化的部分

△ 问题 1: SessionTrace 仅支持事件，不支持持续时间

现状: SessionTrace 只有 record(name, detail) 方法，不支持 "开始→结束" 的时间段追踪。模式切换、镜头切换等需要手动计算两个事件的时间差。

建议优化:

interface SessionTrace {
    fun record(name: String, detail: String)
    fun begin(name: String): TraceHandle    // 新增
    fun end(handle: TraceHandle, detail: String = "")  // 新增
    fun snapshot(): List<SessionTraceEvent>
}

data class TraceHandle(
    val name: String,
    val startNanos: Long,
    val sequence: Int
)

收益: 模式切换、镜头切换、后处理链等耗时追踪可统一用 begin/end 模式，不再需要手动配对事件。

△ 问题 2: timing 数据以字符串形式传递

现状: pipelineNotes: List<String> 是主要的诊断数据载体，timing 信息以 "timing:device=42ms" 格式嵌入字符串列表。

问题:

  • 无法被类型安全地消费
  • 解析需要字符串操作
  • 无法做结构化聚合分析

建议优化: 将 timing 扩展到 ShotResult 的结构化字段（已有 ShotTiming 但未充分利用），并在 pipelineNotes 之外增加 structuredMetrics: Map<String, Any>。

△ 问题 3: 无持续性指标存储

现状: InMemorySessionTrace 是纯内存的 200 事件环形缓冲区，进程退出即丢失。DevLogExporter 仅支持手动导出。

建议优化: 增加轻量级的性能指标持久化（如 SharedPreferences 存储最近 N 次拍摄的 timing 统计），支持跨 session 的性能趋势分析。

修改文件: core/media/MediaPostProcessors.kt

// CompositeMediaPostProcessor.process()
override suspend fun process(result: ShotResult): ShotResult {
    var current = result
    val processorTimings = mutableListOf<Pair<String, Long>>()
    for (processor in processors) {
        val start = SystemClock.elapsedRealtime()
        current = processor.process(current)
        val elapsed = SystemClock.elapsedRealtime() - start
        processorTimings.add(processor::class.simpleName!! to elapsed)
    }
    val timingNotes = processorTimings
        .filter { it.second > 5 } // 只记录超过 5ms 的
        .map { "timing:processor:${it.first}=${it.second}ms" }
    return current.appendNotes(timingNotes)
}

6.3 架构优化方案 (可与 P0/P1 并行)

6.3.1 SessionTrace begin/end 扩展

修改文件: core/session/SessionTrace.kt

data class TraceHandle(
    val name: String,
    val startNanos: Long,
    val sequence: Int
)

interface SessionTrace {
    fun record(name: String, detail: String)
    fun begin(name: String): TraceHandle
    fun end(handle: TraceHandle, detail: String = "")
    fun snapshot(): List<SessionTraceEvent>
}

class InMemorySessionTrace(
    private val maxEvents: Int = 200
) : SessionTrace {
    // ... existing implementation ...

    override fun begin(name: String): TraceHandle {
        val seq = sequence.incrementAndGet()
        synchronized(events) {
            events += SessionTraceEvent(
                sequence = seq,
                name = "$name.started",
                detail = "",
                timestampMillis = System.currentTimeMillis()
            )
            if (events.size > maxEvents) events.removeAt(0)
        }
        return TraceHandle(name, System.nanoTime(), seq)
    }

    override fun end(handle: TraceHandle, detail: String) {
        val elapsedMs = (System.nanoTime() - handle.startNanos) / 1_000_000L
        synchronized(events) {
            events += SessionTraceEvent(
                sequence = sequence.incrementAndGet(),
                name = "${handle.name}.completed",
                detail = if (detail.isEmpty()) "${elapsedMs}ms" else "$detail,${elapsedMs}ms",
                timestampMillis = System.currentTimeMillis()
            )
            if (events.size > maxEvents) events.removeAt(0)
        }
    }
}

八、总结与行动建议

关键发现

  1. 预览和拍摄管道的性能监控已相当完善——首帧延迟有预算评估，拍摄有 4 点计时，热状态和资源预算系统设计良好。
  2. 模式切换和镜头切换是最大的监控盲区——这两个高频操作完全没有耗时统计。
  3. 后处理链缺乏细粒度监控——7 个后处理器只有总耗时，无法定位瓶颈。
  4. 架构层面需要 begin/end trace 能力——当前只有点事件，不支持时间段追踪。
  5. 无基准测试基础设施——没有 AndroidX Benchmark、Baseline Profile 或 Perfetto 集成。

建议行动顺序

  1. Phase 1 (本周): 补齐 G1/G2/G3 三个高优先级缺口 + SessionTrace begin/end 扩展
  2. Phase 2 (下周): 补齐 G4/G5/G8 + timing 结构化
  3. Phase 3 (月内): 预览 FPS 监控 + 算法队列深度 + 跨 session 持久化
  4. Phase 4 (季度): AndroidX Benchmark + Baseline Profile + Perfetto 集成
