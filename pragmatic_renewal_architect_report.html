4. 主导约束判断

主导约束: app 模块 UI 渲染面过度集中——SessionUiRenderModel (2730行) 作为"上帝渲染对象"承载全部面板的 render model 函数，与 MainActivity.render() (736行Activity中的~200行渲染调用) 紧紧耦合。每个 UI/交互需求变更都必然穿越这组文件，它们占据了变更频次的前两名，且 bug 修复密集(状态同步、旋转遗漏、面板切换串位)。

为什么这不是"最明显的重构"而是"真正约束能力的瓶颈"?

  • 变更摩擦: 过去50次提交中约30%直接涉及 UI 渲染面(MainActivity + SessionUiRenderModel + CockpitSurfaceRenderer + activity_main.xml)。新增任何面板或交互(如 ColorLab)都需要在这组文件中"接线"——render model函数、renderer调用、router命令、XML布局。
  • 测试隔离差: SessionUiRenderModelTest 1929行测试与 SessionUiRenderModel 2730行实现强绑定。修改一个面板的 render model 可能无意中破坏另一个面板的测试断言。
  • 后续阻塞: 产品 2.0 标准中还有大量 UI 需求(水印选择器、滤镜调节面板、专业模式 controls、多语言文案)。在 render model 单体不拆的情况下，每个新需求都会继续扩大这组文件。

为什么不是 DefaultCameraSession 拆分更优先?

Session 内核拆分已经在进行中且有明确的 5 个子任务路径。它应该继续，但不应该是"第一优先"，因为: (1) session 内核的拆分不影响 UI 迭代速度——那是另一个维度的瓶颈; (2) session 拆分的稳定性风险高于 UI render model 拆分(涉及状态机行为变更); (3) RI render model 拆分的收益更直接(每次UI变更都受益)且风险更低(纯展示层重组)。

受影响的关键流

  • 面板切换: MainActivity → CockpitPanelRouter → render() → 各 renderer → SessionUiRenderModel
  • 设置变更: SettingsPanelRenderer → SettingsPageRenderModel → SessionSettingsManager → state 回流 → render()
  • 滤镜调节: FilterLabPanelRenderer → FilterLabPageRenderModel → palette touch → apply → render()
  • 缩略图更新: state.presentation → thumbnailUri → render() → ImageView
  • 旋转适配: CameraOrientationMonitor → dispatch(OutputRotationChanged) → render() → 所有控件旋转

不作为的机会成本

如果在当前状态下继续叠加 2.0 需求，SessionUiRenderModel 将突破 3500+ 行，MainActivity 中 render() 方法调用将超过 30+ 个独立渲染调用，bug 修复率可能从当前的 ~40% 进一步升高。

置信度与证伪条件

置信度: 高。基于仓内代码度量(git churn + 行数 + 调用链分析)。
证伪条件: 如果能证明当前 UI 需求不再大幅增长，或新增面板能够在不编辑 SessionUiRenderModel 的情况下独立接入，则此约束不成立。当前代码证据不支持此假设。

8. 自我审查

未读取/未验证的区域

  • 所有 feature/mode-* 插件的完整实现细节 (仅读取了行数和 git churn)
  • core/media 中后处理器 (PhotoWatermarkPostProcessor 887行, PhotoAlgorithmPostProcessor 679行等) 的内部逻辑
  • CameraXCaptureAdapter 的完整 use case binding 代码
  • scripts/ 目录下的验证脚本完整逻辑
  • codex/ 中除 plan.md 和 documentation.md 外的其他文档

基于推断而非事实的结论

  • Stage 7 的 "80%" 进度估算来自 documentation.md 自述，未经独立验证
  • 真机行为的风险等级 (R3 stability risk=3) 是推断——没有真机数据就无法确认风险大小
  • 模式插件独立测试缺失的业务影响是推断——plugin 数量少(7个)且功能相对稳定，可能当前优先级不高

需要用户提供的业务约束

  • 产品发布的时间线和质量标准 (2.0 何时需要 ship?)
  • RAW/DNG, Night multi-frame, Live photo 的产品决策
  • 真机设备的可用性和目标设备型号矩阵
  • 团队规模和成员专长分布
  • 是否有其他并行项目竞争工程资源

本轮分析最可能错误的地方

  • 过度强调 UI 渲染面拆分: 如果后续 UI 需求放缓(产品 2.0 已基本完成)，而 session 稳定性问题在真机上更紧迫，则 R2 可能应排在 R1 前面。
  • 低估了真机验证缺口: 如果 provider death / thermal / long-run 行为在真机上暴露出严重问题，可能需要回到 Stage 7 加固而非继续 UI 工作。
  • 模式插件测试的优先级: 如果各模式行为已稳定(Stage 6 冻结)，添加测试的边际收益可能低于推进 Stage 7 完成。
