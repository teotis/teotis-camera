OpenCamera 成片画质与高像素能力结构诊断

本报告聚焦“成片画质比预览差”和“vivo X300 快捷像素长期卡在约 13MP”两个用户直觉上应当清晰的问题。结论很直接：这里确实不是单点 UI bug，而是静态拍摄质量事实没有被建模成一个可投影、可验证、可降级的契约。

1. 判断摘要

13MP当前用户观测到的快捷像素上限量级
200MPvivo 官方 X300 主摄标称像素级别
LATENCY当前静态拍摄默认质量优先级
5+保存链路可能经过的 JPEG 后处理阶段
最可能的产品级根因: 快捷画质入口被移除后，底层默认仍是 StillCaptureQualityPreference.LATENCY，CameraX 因此走 CAPTURE_MODE_MINIMIZE_LATENCY。这与用户早前“删去快捷中的画质，默认最高画质”的需求相反。
13MP 的最可能技术根因: 能力枚举只从普通 SCALER_STREAM_CONFIGURATION_MAP.getOutputSizes(ImageFormat.JPEG) 取尺寸，然后经过 4:3 过滤和 LARGE_12MP preset 选择。当前代码没有消费最大分辨率 stream map，也没有把高像素能力建模为 supported/degraded/unsupported。
架构判断: 这类问题在一个好框架里应当清晰，因为“用户想要最高画质”和“设备能否给出高像素成片”本质是同一个 still-capture recipe 的不同投影。当前代码把这些事实分散在 session、mode graph、device graph、CameraX adapter、quick UI、postprocessor、diagnostics 和真机验收文档里，所以小修容易只修到某个投影。

3. 根因分层

A. 已能确认的直接缺陷

  1. 默认质量优先级错误: 用户要求“去掉快捷画质，默认最高画质”，但底层默认还在 LATENCY。
  2. 高像素尺寸枚举不完整: 当前只看普通 stream map 的 JPEG output sizes，没有合并 high-resolution sizes 和 maximum-resolution stream map。
  3. 默认分辨率语义仍是 12MP: LARGE_12MP 不是“最大画质”或“最高像素”，而是一个旧 preset。
  4. 保存链路存在重复 JPEG 重编码风险: 只要多个后处理器命中，成片可能经历多次有损重编码。

B. 需要真机证据确认的部分

  1. vivo X300 的高像素尺寸是否出现在普通 stream map、high-resolution output sizes、maximum-resolution stream map，还是仅由厂商私有模式暴露。
  2. CameraX 当前版本是否能真正 bind 某个高像素尺寸；AndroidX 文档说明 PREFER_HIGHER_RESOLUTION_OVER_CAPTURE_RATE 可使用普通和 high-resolution output sizes，但不允许 app 选择 maximum-resolution pixel-mode 下的 ultra high resolutions。
  3. 保存出的 JPEG 实际像素、EXIF、文件大小、处理 notes 和用户看到的 Quick label 是否一致。

C. 不应承诺的部分

不能承诺 OpenCamera 通过标准 CameraX 单帧 JPEG 后处理达到 vivo 系统相机的 ISP、V3+ 影像芯片、多帧 HDR、时域降噪和厂商场景识别水平。可以承诺的是：在标准 API 可达范围内选择最高质量优先级、尽可能发现和绑定高像素尺寸、诚实展示 degraded/unsupported，并避免应用自己的后处理链劣化成片。

结构压力表

压力点表现为什么小修容易失败
缺少一等 still quality contract quality、resolution、outputSize、render recipe、postprocess 分属不同对象。 改 Quick UI 不会自动改 CameraX capture mode；改枚举不一定改默认选择。
capability 没有 provenance 尺寸只是一组 StillCaptureOutputSize，不知道来自 standard/high/maximum，也不知道 bind proof。 无法向用户解释“硬件可能支持但标准路径不可用”。
预览与保存不是同一路径 Preview path 用 matrix/tint/overlay，saved path 用 bitmap postprocess。 用户看到的预期和最终 JPEG 不由同一 recipe 约束。
流程状态漂移 旧 orchestration 的 INDEX、state.tsv、status 文件对完成状态不一致。 研究结论没有转成代码验收门禁，后续 agent 容易重复或误判。

5. 建议抽象: Effective Still Capture Recipe

核心不变量: App 任何地方显示或执行“最高画质/高像素/预览可信”的时候，都必须来自同一个 EffectiveStillCaptureRecipe。如果这个 recipe 不能证明某项能力，就只能显示 degraded/unsupported，而不能静默退到 12MP 或低延迟。

建议数据形状

data class EffectiveStillCaptureRecipe(
    val lensFacing: LensFacing,
    val lensNode: LensNode?,
    val qualityPriority: StillQualityPriority, // QUALITY by default for still photo
    val selectedOutput: StillOutputSelection,
    val bindability: CapabilitySupport,        // SUPPORTED / DEGRADED / UNSUPPORTED
    val previewFidelity: PreviewFidelity,      // EXACT / APPROXIMATE / DEGRADED
    val renderRecipe: RenderRecipe,
    val postprocessPlan: PostprocessPlan,
    val diagnostics: StillQualityDiagnostics
)

投影规则

投影由 recipe 产生什么验收条件
Quick UI 当前像素 label、是否可切换、降级说明。 不能显示 50MP 除非 recipe 选中的 output 真实为约 50MP。
DeviceGraphSpec qualityPreference=QUALITY、明确 output size、CameraX bind 参数。 默认拍照模式必须映射到 CAPTURE_MODE_MAXIMIZE_QUALITY，除非 recipe 显式选择速度优先。
ShotRequest metadata selected width/height、quality priority、capability source、degraded reason、render recipe hash。 保存 JPEG 后可从 notes/EXIF/diagnostics 对账。
Postprocess chain 是否需要 final output pass；编码质量；是否可合并为一次 bitmap write。 多处理器命中时避免多次 JPEG 有损重编码。
Preview exact/approx/degraded 的预览 fidelity。 用户能知道预览只是方向近似，或者看到与保存同 recipe 的可接受近似。
真机 QA 可导出的 probe: stream-map source、bind request、saved dimensions、JPEG notes。 每次“修复高像素/画质”都必须带真机或诚实 blocked 证据。

候选方案竞争

方案能修什么不能修什么判断
只改默认 LATENCY -> QUALITY 立刻改善“默认最高画质”违背点。 不解决 13MP、预览/保存分歧、后处理重编码。 必须做，但不是完整解
只补 high-resolution sizes 枚举 可能让 Quick 出现更高 MP 候选。 不保证 default 选中最大，不保证 CameraX 可 bind maximum-resolution pixel mode。 必须做，但需要 proof
vivo 专用 hardcode 可能在单机型上骗过 label。 破坏能力真实性，无法保证保存尺寸。 拒绝
Effective Still Capture Recipe 同时约束默认质量、尺寸选择、UI、CameraX、保存、预览 fidelity、诊断。 需要分阶段迁移，不能一次性重写。 推荐

6. 落地路线

阶段 0: 立即止血

  1. 把 still photo 默认质量统一改为 QUALITY，覆盖 DefaultCameraSession、StillCaptureConfig、DeviceGraphSpec.stillCapture()、mode runtime defaults，并加回归测试证明 Quick 没有质量行时仍默认最高画质。
  2. 增加测试：初始 session 的 activeDeviceGraph.stillCapture.qualityPreference 必须是 QUALITY；CameraX builder 必须收到 CAPTURE_MODE_MAXIMIZE_QUALITY。
  3. 把后处理 diagnostics 暴露出“处理器列表 + 重编码次数 + 最终 JPEG 质量参数”。

阶段 1: 高像素能力模型

  1. 在 device adapter 枚举合并 ordinary output sizes、getHighResolutionOutputSizes(ImageFormat.JPEG)、以及 Android 12+ maximum-resolution stream map 的 JPEG sizes。
  2. 为 StillCaptureOutputSize 或旁路 metadata 增加 source: standard/high-resolution/maximum-resolution，以及 bind status。
  3. 把默认 still output selection 从 LARGE_12MP 改成“最高可支持 output”或“用户选择的 output”，保留 12MP 作为兼容降级而不是“最大画质”。
  4. Quick label 继续按实际 pixel count 展示，但必须能显示 degraded badge 和原因。

阶段 2: Recipe 投影

  1. 先引入只读 EffectiveStillCaptureRecipe builder，不立刻重排全部 pipeline。
  2. 让 Quick UI、DeviceGraphSpec、ShotRequest metadata、Dev probe 都从 builder 输出读取。
  3. 将 postprocessor 从“串行有损重编码”逐步改为“decode once, apply planned operations, encode once”。
  4. 把 preview fidelity 明示到 render model：exact / approximate / degraded，避免用户把近似预览当成最终成片承诺。

阶段 3: 真机闭环

  1. 在 vivo X300 上导出 camera probe：camera id、physical id、lens node、所有 JPEG size source、CameraX bind target、saved JPEG dimensions。
  2. 拍摄至少 3 组样张：普通拍照中性、Color Lab 非中性、水印/画幅命中后处理。
  3. 每组记录 Quick label、预览截图、保存 JPEG、EXIF、diagnostics notes，形成 pass/degraded/blocked 判断。

真机门禁

验收项必须采集失败时结论
高像素发现 普通 output sizes、high-resolution output sizes、maximum-resolution stream map sizes。 没有高像素 source: unsupported；有但不能 bind: degraded。
高像素保存 Quick label、bound target、保存 JPEG 实际宽高、EXIF。 label 与保存不一致即失败。
成片不劣化 原始 CameraX JPEG 大小、后处理后 JPEG 大小、重编码次数、样张。 后处理重编码导致明显劣化即失败或 degraded。
预览可信 同场景预览截图、保存图、recipe/fidelity notes。 若 preview 只是 approximate，UI/Dev 必须诚实标注。

8. 外部参考

这些参考只用于确定平台能力边界，不替代真机证明。

  • Android StreamConfigurationMap: 普通 output sizes 与 high-resolution output sizes 是不同查询面。
  • Android CameraCharacteristics maximum-resolution stream map: maximum-resolution pixel mode 有独立 stream configuration map，且可为空。
  • CameraX ImageCapture capture modes: maximum-quality 优先图像质量，minimum-latency 优先速度且可能降低质量。
  • CameraX ResolutionSelector: higher-resolution mode 会看 ordinary 与 high-resolution output sizes，但文档明确不允许 app 选择 maximum-resolution pixel-mode 的 ultra high resolutions。
  • vivo X300 Global: 官方页面标称 200 MP ZEISS 主摄、50 MP 长焦、50 MP 超广角、50 MP 前摄。
  • vivo X300 中国参数页: 官方参数列出 2 亿像素蔡司超级主摄、5000 万像素长焦和超广角。
  • Apple AVFoundation quality prioritization: Apple 也把 speed / balanced / quality 做成显式优先级。
  • Apple AVCapturePhotoSettings maxPhotoDimensions: Apple 的高分辨率选择要求 requested dimensions 与 active format 支持的 dimensions 对齐。
