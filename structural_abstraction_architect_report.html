OpenCamera 结构抽象分析报告

Structural Abstraction Architect | 2026-06-02 | main branch

1. 执行摘要

OpenCamera 是一个分层架构的 Android Kotlin 相机应用，采用 app + 8 个 core 模块 + 8 个 feature 模块的多模块结构。整体架构方向正确（Session Kernel + Mode Plugin + Device Adapter + Media Pipeline），但经过全面分析发现了 6 个主要结构压力点，其中 3 个具有高杠杆价值。

最高杠杆机会

  • P1: 静态拍摄模式基类提取 — Photo/Night/Portrait/Humanistic 共享 ~70% 相同模式结构代码，可删除 ~1200 行重复
  • P2: CameraXCaptureAdapter 拆分 — 3437 行 God Class 应拆分为 4-5 个职责明确的子组件
  • P3: 水印样式集合化 — 6 个模板 × 4 个字段 = 24 个序列化键，应替换为 Map 结构

拒绝的候选方案

  • DeviceEvent/SessionIntent 合并 — 1:1 映射是有意的层级隔离，合并会破坏架构边界
  • SessionState 合并 PresentationState — 分离是有意的关注点分离设计

5. 提案 P1: 静态拍摄模式基类

P1: 提取 AbstractStillCaptureMode 基类 已验证 杠杆:高 负担:低

透镜

Patch Pressure Latent Kernel

症状

PhotoModePlugin:493, PortraitModePlugin:623, NightModePlugin:490, HumanisticModePlugin:513

  • 4 个插件共享 MutableStateFlow、buildSnapshot、FrameRatioDelegate、watermark 工具函数、buildEffectSpec 模式、CaptureStrategy 构建模式
  • toCaptureSpec() 在 3 个文件逐字复制
  • reduceStillShotSessionEvent 调用模式完全一致

不变量假说

这些模式都是"静态拍摄模式"变体：接收 Intent → 构建 EffectSpec → 组装 CaptureStrategy → 处理 SessionEvent → 管理 Snapshot。差异仅在滤镜选择、额外 Effect、Flash、ProVariant。

保持的差异

  • Photo: Flash、LowLight 夜景辅助
  • Portrait: PortraitEffect、ProVariant、风格选择器
  • Night: 多帧合成 NightProfile 系统
  • Humanistic: 人文滤镜选择器、ProVariant

7. 提案 P3: 水印样式集合化

P3: 水印模板样式从 6 字段改为 Map 已验证 杠杆:中 负担:低

症状

PhotoSettings: 6 个 WatermarkStyleSettings 字段, Serializer: 24 个 KEY 常量

PhotoSettings 为每个水印模板存储独立的 WatermarkStyleSettings 字段。watermarkStyleFor() 和 updateWatermarkStyle() 使用 when 链匹配。添加新模板需要：修改 data class + 修改 when 链 + 添加 4 个 serializer KEY + 修改 toMap/fromMap。

建议

// 之前: 6 个字段
data class PhotoSettings(
    val classicOverlayWatermarkStyle: WatermarkStyleSettings = ...,
    val travelPolaroidWatermarkStyle: WatermarkStyleSettings = ...,
    // ... 4 more
)

// 之后: 1 个 Map
data class PhotoSettings(
    val watermarkStyles: Map<String, WatermarkStyleSettings> = DEFAULT_WATERMARK_STYLES
) {
    fun watermarkStyleFor(templateId: String): WatermarkStyleSettings =
        watermarkStyles[templateId] ?: DEFAULT_WATERMARK_STYLE
}

11. 拒绝/推迟的抽象

R1: DeviceEvent/SessionIntent 统一 拒绝

理由: CameraSessionCoordinator 的 1:1 映射是有意的层级隔离设计。DeviceEvent 来自硬件层，SessionIntent 来自应用层语义。合并会破坏架构边界，使 Session Kernel 依赖硬件实现细节。当前映射代码虽然机械但清晰、无 bug、不需维护。

R2: SessionEffect/DeviceCommand 统一 拒绝

理由: 同 R1。SessionEffect 携带 ModeId、DeviceGraphSpec 等会话语义，DeviceCommand 是纯硬件指令。Coordinator 的转换逻辑包含了 syncActiveDeviceCapabilities 等附加行为，不是纯 1:1 映射。

R3: SessionState 合并 PresentationState 拒绝

理由: SessionPresentationState 包含 UI 特有的 transient 状态（缩略图路径、倒计时秒数、预览快照代数），与 SessionState 的运行时不变量有意分离。合并会导致 Session Kernel 需要了解 UI 演示细节。

R4: Video 模式纳入静态拍摄基类 误报

理由: VideoModePlugin 的行为模式完全不同：不使用 SaveRequest photoLibrary、不使用 FrameRatioDelegate、不使用 WatermarkEffect、有 Recording 状态机。强行纳入基类会增加异常数量而非减少。

12. 过渡交接简报

推荐执行顺序

  1. P3 水印集合化（独立、低风险、可验证）— 先做，为 P5 提供数据
  2. P1 模式基类（最高杠杆、渐进迁移）— 从 PhotoModePlugin 开始
  3. P6 元数据 Builder（P1 的自然延伸）
  4. P4 PostProcessor 桥接统一（独立、低风险）
  5. P5 Serializer（P3 后重新评估）
  6. P2 Adapter 拆分（最高迁移负担，放最后）

安全约束

  • 每个提案独立可回滚
  • 不修改公共 API 边界（CameraDeviceAdapter、CameraSession）
  • 每个迁移步骤运行 Stage 7 验证脚本
  • P2 需要最多测试覆盖，建议在其他提案完成后执行
