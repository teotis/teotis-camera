    value: String?

具体症状

6 个 PostProcessor 实现 (PhotoFrameRatioPostProcessor, PhotoSelfieMirrorPostProcessor, DocumentAutoCropPostProcessor, PortraitRenderPostProcessor, PhotoAlgorithmPostProcessor, PhotoWatermarkPostProcessor) 都遵循相同的三态工作判定模式:

1. sealed interface XxxWork { None; Crop|Mirror|Apply(...); DiagnosticSkip(reason) }
2. sealed interface XxxTarget { FilePath; ContentUri }
3. sealed interface XxxEditorResult { Applied; Skipped; Failed }
4. fun decideXxxWork(result: ShotResult): XxxWork — 检查 mediaType/mimeType/metadata/target
5. interface XxxEditor { suspend fun apply(target): XxxEditorResult }

隐藏不变量假设

所有图片后处理器共享一个潜在协议: "检查 ShotResult 是否需要本处理器的工作；如果需要，提取目标；调用平台编辑器执行；报告结果到 pipelineNotes"。这不是偶然的结构相似——它们都服务于同一个 ShotResult → ShotResult 的转换管道。

有意义的差异保留

各处理器的工作判定条件不同(有的检查 frameRatio，有的检查 selfieMirrorApply，有的检查 watermark template)。目标类型不同(FilePath vs ContentUri vs Bitmap)。编辑器接口不同。规范化协议只统一判定框架和结果报告，不统一判定条件和编辑器接口。

5. 被拒绝或延迟的抽象

False Abstraction Risk RA-1: 为 SessionIntent / SessionEffect / DeviceCommand 引入泛型消息总线

症状: SessionIntent 有 20+ 个子类型，SessionEffect 有 7 个子类型，DeviceCommand 有 6 个子类型。看起来可以统一为泛型消息总线。
拒绝原因: 这些类型的分离是有意义的域边界——SessionIntent 是 UI→Session 的单向通道，SessionEffect 是 Session→Coordinator 的单向通道，DeviceCommand 是 Coordinator→Adapter 的单向通道。统一它们会消除类型安全，不会删除任何实际复杂度。
结论: 这是一个名称统一但语义分离的假抽象。

False Abstraction Risk RA-2: 为所有 RenderModel 引入统一的 RenderModel 接口

症状: 56 个 render model data class 看起来可以统一。
拒绝原因: 这些 data class 的值在于它们的具体字段——每个面板需要不同的数据。统一接口只能提供空洞的 "is a RenderModel" 标记，不会删除任何分支、适配器或重复逻辑。
结论: 这是框架化而非结构化——引入一个 DSL/接口层不等于简化。

False Abstraction Risk RA-3: 合并 FeatureCatalogAction 和 PersistedSettingsAction

症状: 两套平行的 action/reduce 系统。
拒绝原因: 它们操作不同的状态对象 (FeatureCatalog vs PersistedSettings)，有不同的持久化语义 (catalog 不持久化用户数据，settings 持久化)，有不同的生命周期。合并会引入不必要的耦合。
结论: 这是有意义的边界——两个独立的设置域，不应强制统一。

7. 不确定性和缺失证据

缺失证据:

  • 运行时性能: 没有 profiling 数据。泛型 fromStorageKey 的内联行为需要在目标设备上验证。
  • 模式插件差异完整度: 只读取了 PhotoModePlugin 的完整实现，其他 6 个模式只看了行数和 import。P6 的置信度依赖于对所有模式的完整对比。
  • PostProcessor EditorResult 一致性: 只读取了 PhotoSelfieMirrorPostProcessor 和 PhotoFrameRatioPostProcessor 的完整实现。P2 的置信度依赖于确认所有 6 个处理器的 Applied/Skipped/Failed 语义完全一致。
  • 未来需求方向: 如果产品 2.0 需要更多水印模板或新的 settings 域，P5 的价值会上升。如果模式数量稳定在 7 个，P6 的价值会下降。

本轮分析最可能错误的地方:

  • 过度高估 P1 的价值: StorageKey 枚举样板虽然冗长但完全正确。100 行的减少可能不值得引入新的接口层——如果团队对当前模式已经习惯，"更紧凑"不等于"更好"。
  • 低估 P2 的过渡负担: 泛型 ProcessorWork 协议需要修改 6 个 PostProcessor 的内部实现。虽然接口不变，但每个处理器的 decideWork 函数需要适配新的泛型类型——这可能比看起来更复杂。
  • P4 被过早标记为"务实排序": Settings Action/Reduce 模板化可能实际上比 P1/P2 更有价值——因为它影响的是整个设置系统的可扩展性，而不仅仅是样板减少。
